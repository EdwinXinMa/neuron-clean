一、概述
这份文档是做什么的

1.1 文档说明

本文档定义了 N3 Lite 设备端与云平台之间的通信协议。硬件开发人员根据本文档实现 N3 Lite 的云端通信功能。

阅读对象：N3 Lite 硬件/固件开发人员
核心原则：每个消息都给出完整的 JSON 示例，照着示例开发即可。

1.2 系统架构

N3 Lite 作为虚拟充电桩通过 OCPP 1.6-J 协议连接云平台。下挂的 ATP III 充电桩不直接与云平台通信，所有数据经 N3 Lite 代理上报。

                    WiFi / 互联网
  N3 Lite  ──────────────────────────────  云平台
    │              WebSocket (OCPP 1.6-J)
    │ PLC
    ├── ATP III 桩1（枪1、枪2）
    └── ATP III 桩2（枪1、枪2）
关键概念：N3 Lite 在 OCPP 协议中扮演 Charge Point（充电桩）角色，云平台扮演 Central System（中央系统）角色。N3 Lite 下挂的每把充电枪映射为 OCPP 的 Connector。

1.3 优先级说明

优先级功能说明
第一优先 设备注册、心跳、拓扑上报、拓扑变更、状态/告警上报、DLM 运行数据上报 连上来、看得到 — 云平台的基础能力
第二优先 远程 DLM 配置、N3 Lite OTA 升级 能远程操作 — 解决售后问题
第三优先 充电启停、充电记录、远程启停、下挂桩 OTA 预留，后续版本实现
二、通信基础
怎么连、消息长什么样

2.1 连接方式

项目说明
协议WebSocket
子协议ocpp1.6（WebSocket 握手时在 Sec-WebSocket-Protocol 头中指定）
连接地址ws://{服务器地址}:{端口}/ocpp/{设备序列号}
示例ws://cloud.alwayscontrol.com:9001/ocpp/NL-20260301-0001
端口9001
重要：URL 路径中的 {设备序列号} 就是 N3 Lite 的唯一标识（SN），云平台通过它识别是哪台设备。每台 N3 Lite 使用自己的 SN 连接，不能共用。

连接与重连

  • 首次连接：N3 Lite 通电并连上 WiFi 后，立即发起 WebSocket 连接
  • 断线重连：连接断开后自动重连，间隔递增 — 1s → 2s → 4s → 8s → 最大 30s
  • 连接成功后：立即发送 BootNotification（见第三章）

2.2 消息格式

OCPP 1.6-J 使用 JSON 数组格式，共三种消息类型：

请求（CALL）— 发起方发送

[2, "{唯一消息ID}", "{动作名称}", {载荷}]

示例：

[2, "msg001", "Heartbeat", {}]

成功响应（CALLRESULT）— 接收方回复

[3, "{对应的消息ID}", {载荷}]

示例：

[3, "msg001", {"currentTime": "2026-03-17T10:00:00Z"}]

错误响应（CALLERROR）— 接收方回复错误

[4, "{对应的消息ID}", "{错误码}", "{错误描述}", {详情}]

示例：

[4, "msg001", "InternalError", "Server error", {}]
消息ID 规则：每次请求使用唯一的消息 ID（建议用递增序号如 msg001、msg002），响应中的消息 ID 必须与请求一致，用于匹配请求和响应。

消息方向说明

标记方向说明
上行N3 Lite → 云平台设备主动发送请求，云平台回复响应
下行云平台 → N3 Lite云平台主动发送请求，设备回复响应

2.3 ConnectorId 映射规则

N3 Lite 作为虚拟充电桩，下挂的充电桩的每把枪映射为一个 ConnectorId：

ConnectorId对应设备说明
0N3 Lite 本体上报 N3 Lite 自身的状态/故障
1桩1 - 枪1第一台 ATP III 充电桩的两把枪
2桩1 - 枪2
3桩2 - 枪1第二台 ATP III 充电桩的两把枪（如有）
4桩2 - 枪2
注意：ConnectorId 的分配在 TopologyReport 中确定。云平台通过 TopologyReport 知道 ConnectorId 1/2 属于哪台桩、3/4 属于哪台桩。后续所有消息（StatusNotification 等）中的 ConnectorId 必须与 TopologyReport 中上报的一致。
三、场景一：设备启动与注册
通电后做什么

第一优先 — N3 Lite 通电联网后的第一件事。

3.1 启动流程

1
连接 WiFi — N3 Lite 通电后连接已配置的 WiFi 网络
2
建立 WebSocket 连接 — 连接到云平台 ws://服务器/ocpp/{SN}
3
发送 BootNotification — 上报设备身份信息，等待云平台回复 Accepted
4
发送 TopologyReport — 上报下挂充电桩和枪的完整信息
5
发送各枪 StatusNotification — 上报每把枪的初始状态
6
开始心跳 — 按照 BootNotification 响应中的 interval 值定时发送 Heartbeat

3.2 BootNotification

上行 OCPP 原生 — 设备注册，上报身份信息

请求（N3 Lite → 云平台）

[2, "msg001", "BootNotification", {
  "chargePointVendor": "AlwaysControl",
  "chargePointModel": "N3 Lite",
  "chargePointSerialNumber": "NL-20260301-0001",
  "firmwareVersion": "2.1.3"
}]
字段类型必填说明
chargePointVendorString是固定值 "AlwaysControl"
chargePointModelString是固定值 "N3 Lite"
chargePointSerialNumberString是N3 Lite 的序列号（SN），与 URL 路径中的一致
firmwareVersionString是当前固件版本号

响应（云平台 → N3 Lite）

[3, "msg001", {
  "status": "Accepted",
  "currentTime": "2026-03-17T10:00:00Z",
  "interval": 60
}]
字段类型说明
statusString"Accepted" 表示注册成功
currentTimeString云平台当前时间（ISO 8601 格式），设备可用于校时
intervalInteger心跳间隔，单位秒。设备按此间隔发送 Heartbeat
处理逻辑：收到 "Accepted" 后，记录 interval 值用于心跳，然后立即发送 TopologyReport。如果收到 "Rejected"，等待 interval 秒后重新发送 BootNotification。

3.3 TopologyReport（自定义）

上行 自定义扩展 — 上报下挂充电桩和枪的完整信息

为什么需要自定义？
OCPP 标准的 Connector 没有序列号和型号的概念。云平台需要知道每把枪对应的是哪台物理充电桩，以及桩的序列号和型号，所以通过 DataTransfer 扩展上报。

请求（N3 Lite → 云平台）

[2, "msg002", "DataTransfer", {
  "vendorId": "AlwaysControl",
  "messageId": "TopologyReport",
  "data": "{JSON 字符串，内容见下方}"
}]

data 字段的 JSON 内容（转义前）：

{
  "chargePoints": [
    {
      "sn": "AT-0001",
      "model": "ATP III-11.5",
      "connectors": [
        { "connectorId": 1, "status": "Available" },
        { "connectorId": 2, "status": "Available" }
      ]
    },
    {
      "sn": "AT-0002",
      "model": "ATP III-11.5",
      "connectors": [
        { "connectorId": 3, "status": "Available" },
        { "connectorId": 4, "status": "Available" }
      ]
    }
  ]
}
字段类型必填说明
chargePointsArray是下挂充电桩列表
chargePoints[].snString是充电桩序列号
chargePoints[].modelString是充电桩型号
chargePoints[].connectorsArray是该桩下的枪列表
connectors[].connectorIdInteger是枪编号（全局唯一，从 1 开始递增）
connectors[].statusString是枪当前状态（见附录 B）

响应（云平台 → N3 Lite）

[3, "msg002", {
  "status": "Accepted"
}]
发送时机：
1. BootNotification 成功后立即发送一次（全量上报）
2. 设备重启后发送一次
3. 拓扑发生变化时发送（见第六章 TopologyChange）
四、场景二：心跳保活
告诉云平台"我还活着"

第一优先 — 云平台通过心跳判断设备是否在线。

4.1 Heartbeat

上行 OCPP 原生

请求（N3 Lite → 云平台）

[2, "msg003", "Heartbeat", {}]

响应（云平台 → N3 Lite）

[3, "msg003", {
  "currentTime": "2026-03-17T10:01:00Z"
}]
发送规则：
• 按照 BootNotification 响应中的 interval 值定时发送（如 interval=60，则每 60 秒发一次）
• 如果在心跳间隔内已经发送过其他消息（如 StatusNotification），可以跳过本次心跳
• 云平台超过 3 倍心跳间隔未收到任何消息，将判定设备离线
五、场景三：状态与告警上报
设备状态变了、出故障了，告诉云平台

第一优先 — 让云平台实时掌握设备状态。

5.1 StatusNotification

上行 OCPP 原生 — 上报设备/枪的状态变化和故障信息

请求（N3 Lite → 云平台）

示例1：枪1 状态变为"充电中"

[2, "msg004", "StatusNotification", {
  "connectorId": 1,
  "status": "Charging",
  "errorCode": "NoError",
  "timestamp": "2026-03-17T10:05:00Z"
}]

示例2：枪2 发生过流故障

[2, "msg005", "StatusNotification", {
  "connectorId": 2,
  "status": "Faulted",
  "errorCode": "OverCurrentFailure",
  "vendorErrorCode": "E1023",
  "info": "Phase L1 over current 35A",
  "timestamp": "2026-03-17T10:06:00Z"
}]

示例3：N3 Lite 本体故障

[2, "msg006", "StatusNotification", {
  "connectorId": 0,
  "status": "Faulted",
  "errorCode": "HighTemperature",
  "info": "Internal temperature 82°C",
  "timestamp": "2026-03-17T10:07:00Z"
}]
字段类型必填说明
connectorIdInteger是0=N3 Lite 本体，1/2/3/4=对应枪号
statusString是当前状态（见附录 B）
errorCodeString是错误码，无错误时填 "NoError"（见附录 A）
vendorErrorCodeString否厂商自定义错误码，用于更细粒度的故障定位
infoString否故障描述文字（最长 50 字符）
timestampString是状态变化时间（ISO 8601 格式）

响应（云平台 → N3 Lite）

[3, "msg004", {}]
发送时机：状态发生变化时立即发送，不需要定时。包括：
• 枪状态变化（空闲 → 充电中、充电中 → 空闲、任意状态 → 故障）
• N3 Lite 本体故障（connectorId=0）
• 故障恢复（status 从 Faulted 变回正常状态，errorCode 改为 NoError）
桩级别故障处理：如果整台充电桩故障（而不是某把枪），将该桩下所有枪都上报为 Faulted。例如桩1整机故障，同时发送 connectorId=1 和 connectorId=2 的 Faulted 状态。

5.2 DLMStatus（自定义）

上行 自定义扩展 — 上报 N3 Lite 的核心运行数据

这是最重要的自定义消息。没有这个数据，云平台只能看到设备在线/离线，看不到设备实际的运行状况。

请求（N3 Lite → 云平台）

[2, "msg007", "DataTransfer", {
  "vendorId": "AlwaysControl",
  "messageId": "DLMStatus",
  "data": "{JSON 字符串，内容见下方}"
}]

data 字段的 JSON 内容：

{
  "breakerRating": 32,
  "totalCurrent": 30.8,
  "voltage": 231.5,
  "totalPower": 7.12,
  "deviceTemp": 42.5,
  "wifiRssi": -55,
  "uptime": 86400,
  "pileAllocations": [
    { "sn": "AT-0001", "allocatedCurrent": 11.0, "actualCurrent": 10.8 },
    { "sn": "AT-0002", "allocatedCurrent": 11.0, "actualCurrent": 10.5 }
  ]
}
字段类型必填单位说明
breakerRatingInteger是A当前断路器档位设置（20/25/32/40/50/63）
totalCurrentNumber是ACT 采集的入户总电流
voltageNumber是V当前电压
totalPowerNumber是kW总功率
deviceTempNumber是°CN3 Lite 内部温度
wifiRssiInteger是dBmWiFi 信号强度（负值，越大越好，如 -55 > -80）
uptimeInteger是秒设备运行时长（从本次通电开始计算）
pileAllocationsArray是各桩的电流分配信息
pileAllocations[].snString是桩序列号
pileAllocations[].allocatedCurrentNumber是ADLM 分配给该桩的电流
pileAllocations[].actualCurrentNumber是A该桩实际使用的电流

响应（云平台 → N3 Lite）

[3, "msg007", {
  "status": "Accepted"
}]
发送时机：
• 定时上报：建议每 30~60 秒上报一次
• 断路器档位变化时立即上报一次
• 可与心跳交替发送（如心跳 60 秒，DLMStatus 在中间 30 秒时发送）
六、场景四：拓扑变更
充电桩加了一台、掉了一台

第一优先 — 运行中桩的增减需要通知云平台。

6.1 TopologyChange（自定义）

上行 自定义扩展

当 N3 Lite 检测到下挂充电桩发生变化（通过 PLC 发现新桩或桩失联），重新发送一次完整的 TopologyReport 即可。

实现方式：不需要单独定义 TopologyChange 消息。拓扑变化时，重新发送一次 TopologyReport（第三章 3.3），云平台会对比新旧拓扑，自动处理桩的增加和删除。

全量覆盖：每次 TopologyReport 都是完整的当前拓扑，不是增量。云平台以最新一次 TopologyReport 为准。

7.1 DLMConfig（自定义）

下行 自定义扩展 — 云平台下发 DLM 配置到 N3 Lite

请求（云平台 → N3 Lite）

[2, "req001", "DataTransfer", {
  "vendorId": "AlwaysControl",
  "messageId": "DLMConfig",
  "data": "{\"breakerRating\": 25}"
}]
字段类型必填说明
breakerRatingInteger是目标断路器档位，可选值：20 / 25 / 32 / 40 / 50 / 63

响应（N3 Lite → 云平台）

成功：

[3, "req001", {
  "status": "Accepted"
}]

充电中拒绝：

[3, "req001", {
  "status": "Rejected",
  "data": "Charging in progress, cannot modify DLM settings"
}]
充电中保护：任意充电桩正在充电时，N3 Lite 必须拒绝 DLM 配置修改，返回 "Rejected"。这与本地蓝牙 App 的行为一致 — 充电中禁止修改最大入户电流设置。
生效确认：N3 Lite 应用新配置后，在下一次 DLMStatus 上报中体现新的 breakerRating 值，云平台以此确认配置已生效。
八、场景六：N3 Lite OTA 升级

8.1 OTA 流程

N3 Lite 自身的 OTA 使用 OCPP 原生的 UpdateFirmware + FirmwareStatusNotification。

1
云平台下发 UpdateFirmware — 告诉 N3 Lite 固件下载地址
2
N3 Lite 通过 WiFi 下载固件 — 从指定 URL 下载 .bin 文件
3
N3 Lite 上报下载进度 — FirmwareStatusNotification(Downloading)
4
下载完成，校验并安装 — FirmwareStatusNotification(Installing)
5
安装完成，设备重启 — 重启后重新连接，发送 BootNotification（新固件版本）

UpdateFirmware（云平台 → N3 Lite）下行 OCPP 原生

[2, "req002", "UpdateFirmware", {
  "location": "https://ota.alwayscontrol.com/firmware/N3Lite-2.2.0.bin",
  "retrieveDate": "2026-03-17T12:00:00Z"
}]
字段类型说明
locationString固件下载 URL
retrieveDateString开始下载的时间（可立即执行，设为当前时间即可）

响应（N3 Lite → 云平台）

[3, "req002", {}]

FirmwareStatusNotification（N3 Lite → 云平台）上行 OCPP 原生

[2, "msg008", "FirmwareStatusNotification", {
  "status": "Downloading"
}]
status 值说明
Downloading正在下载固件
Downloaded下载完成
Installing正在安装
Installed安装成功（随后设备会重启）
DownloadFailed下载失败
InstallationFailed安装失败（自动回滚到上一版本）
充电中保护：任意充电桩正在充电时，N3 Lite 不应立即执行 OTA。应等待所有充电结束后再开始下载和安装。设备可以先记录下升级指令，待充电结束后自动执行。

对照表

A. OCPP 错误码对照表

StatusNotification 中 errorCode 字段的可选值：

errorCode含义说明
NoError无错误正常状态，必须填此值
GroundFailure接地故障检测到接地异常
HighTemperature过温设备温度过高
OverCurrentFailure过流电流超过安全阈值
OverVoltage过压电压过高
UnderVoltage欠压电压过低
InternalError内部错误设备内部故障
PowerSwitchFailure开关故障继电器/接触器故障
ConnectorLockFailure枪锁故障充电枪锁定/解锁失败
OtherError其他错误不属于以上类别的错误，用 vendorErrorCode 和 info 补充说明

B. 枪状态对照表

StatusNotification 和 TopologyReport 中 status 字段的可选值：

status中文含义说明
Available空闲可以使用，没有充电
Preparing准备中充电枪已插入，等待授权或启动
Charging充电中正在充电
SuspendedEV车辆暂停车辆主动暂停充电
SuspendedEVSE桩端暂停桩端暂停充电（如 DLM 限流到 0）
Finishing结束中充电完成，等待拔枪
Faulted故障设备故障，不可使用
Unavailable不可用设备维护或被禁用
