一、为什么做
项目背景与动机

我们为什么要做这个云平台？不做会怎样？做了能带来什么？

业务背景

现状: 每台设备都是一座孤岛

智能网关 (N3 Lite / N3)。这些设备安装在用户家中或商业场所，但目前它们有一个共同的问题: 卖出去之后，我们和设备之间就"失联"了。

现状痛点具体影响业务代价
设备离线用户出门后无法知道充电进度，不能远程启停用户体验差，产品竞争力不足
无法远程诊断设备报故障，售后团队无从判断原因每一次售后都是上门服务，成本高、响应慢
数据无沉淀不知道设备运行多少小时、充了多少电、什么时候出过故障产品迭代缺乏数据支撑，用户也没有充电记录可查
固件无法远程更新发现Bug或要加新功能，需要上门刷机新功能推送周期长，Bug修复成本高
一锤子买卖硬件卖出即结束，没有持续连接无法构建充电服务生态，错失未来运营收入
核心问题: 没有云平台，AlwaysControl 只是一个"卖硬件的公司"。设备交付之后，与用户的关系就断了。我们对设备的运行状态一无所知，用户也享受不到智能化的充电体验。在充电桩行业竞争日益激烈的今天，没有云平台的充电产品已经不具备竞争力。

目标: 让每台设备"永远在线"

NeuronCloud 的核心使命只有一句话: 把每一台离线的充电设备变成在线的、可管的、有数据的智能终端。

受益方有了云平台能做什么商业价值
终端用户远程监控充电状态、查看历史记录、预约充电、电费追踪、故障实时通知提升用户体验，增强产品粘性
AlwaysControl远程诊断设备问题、一键OTA推送固件、全量设备健康监控、用户行为数据分析降低售后成本，驱动产品迭代，建立数据资产
商业运营商 (未来)充电计费结算、多站点管理、收入报表、开放API对接第三方系统开辟 SaaS 运营收入，让软件平台也有自己的盈利能力

商业模式转变

今天的模式: 卖硬件 (一次性收入) → 交付即结束 → 设备变孤岛。

目标模式: 硬件 + 云平台 + APP (持续收入) → 长期服务关系 → 数据驱动迭代。

硬件是入口，云平台是护城河。 每多卖一台设备，平台的数据和网络价值就增加一分。设备越多，平台价值越大，用户越难离开 — 这就是飞轮效应。未来，平台本身可以成为独立的商业产品 (SaaS 充电管理平台)，而不仅是硬件的附属品。
设计理念

五条核心设计理念

在动手设计任何一个模块之前，我们先确定几条贯穿整个平台的设计原则。后续的每一个技术决策，都应该可以追溯到这些理念。

理念一: 双协议并存，统一数据模型

充电桩和网关是两类不同的设备，各自有最适合的通信协议。充电桩用 OCPP — 充电桩行业部署量最大的标准协议，开箱覆盖充电事务全流程，且能接入第三方品牌桩。网关用 MQTT — 轻量灵活，适合网关的自定义数据上报。

两种协议进来，业务层怎么办？
接入层负责把 OCPP 和 MQTT 消息都转换为统一的内部数据格式，投递到 Kafka。业务层 (充电管理、能耗分析、告警) 只消费 Kafka 中的统一格式消息，完全不知道也不需要知道数据来自哪个协议。未来新增一种接入协议 (比如 Modbus TCP)，只需要新增一个接入模块做消息转换，所有业务模块零改动。

理念二: 关键数据零丢失

充电记录、计费数据、告警事件 — 这些数据如果丢失，直接影响用户信任和商业结算。我们在架构的每一层都为数据可靠性设计了保障机制。

为什么不能"先跑通再补可靠性"？
数据可靠性不是"优化项"，而是"基础设施"。一旦上线后丢失用户的充电记录或计费数据，修复信任的成本远高于一开始就做对。而且，我们选择的技术栈(MQTT QoS + Kafka + 业务ACK)本身就内建了可靠性，不需要额外付出太多成本。

理念三: 模块化单体，预留微服务边界

初期所有模块部署在一个JVM进程中，但每个模块有独立的代码边界(Maven module)、独立的数据访问、清晰的接口定义。

为什么不一开始就做微服务？
项目初期设备规模有限，过早引入微服务会带来分布式事务、服务治理、部署复杂度等一系列额外成本，开发效率会显著降低。合理的策略是: 在设计阶段就将模块边界划分清楚，初期以单体部署快速推进，当某个模块真正成为性能瓶颈或团队协作瓶颈时再进行拆分。拆分的改动量可控 (增加启动类 + 替换远程调用)，而过早拆分的代价则难以回收。

理念四: 离线安全，来网即同步

充电桩和网关在断网时必须能独立工作。云端的角色是"管理和增值"，不是"控制和依赖"。

为什么强调离线安全？
WiFi断了、路由器重启、云端升级 — 这些场景在家庭环境中很常见。如果充电桩因为断网就不能充电，那是产品设计缺陷。云端只下发配置和策略，不参与实时控制环路。设备本地有完整的充电控制逻辑和安全保护，断网时正常工作，联网后自动将离线期间的数据补传到云端。

理念五: 分阶段交付，先闭环再扩展

不追求一次性做完所有功能。Phase 1 只做最小闭环: 设备能上线、数据能看到、固件能更新。验证这条路走通了，再做充电管理、能耗分析、计费运营。

为什么不一次性规划完所有功能？
架构上我们规划了完整的 Phase 1-4 路线图，但交付节奏是渐进的。先用最短时间让第一台设备真正上云、数据真正可见，这比在纸面上规划一个完美系统要有价值得多。Phase 1 的成功会验证整个技术栈和数据链路，为后续的功能扩展提供信心和经验。

充电桩为什么选 OCPP，而不是其他充电桩协议？
充电桩行业有多种通信协议可选，我们对比后选择了 OCPP 1.6:
协议特点为什么不选
OCPP 1.6开放标准，全球部署量最大，定义了充电桩完整生命周期交互 (注册/心跳/充电事务/计量/远程控制/OTA)，基于 WebSocket/JSON，开发友好选它
私有协议自定义 TCP/HTTP，完全自由需要从零设计充电桩通信规范，工作量大；第三方充电桩无法接入；协议维护成本全部自己扛
OCPP 2.0.1OCPP 的新版本，安全性和设备管理增强市场普及度远低于 1.6，大量充电桩尚未支持。架构设计上预留 2.0.1 扩展能力，未来可平滑升级
IEC 63110IEC 新一代充电管理标准仍在制定中，尚未形成商用生态，目前没有桩厂商支持
OpenADR需求响应协议侧重电网调度和需求响应，不覆盖充电桩日常管理 (注册/事务/计量等)，定位不同
核心理由: OCPP 1.6 是当前充电桩管理领域部署量最大、生态最成熟的协议。充电桩的注册、充电事务、计量上报、远程控制、固件更新 — 全部开箱即用，不需要我们从零设计。更重要的是，市场上大量第三方充电桩已内置 OCPP 支持，采用它意味着平台不会与自有硬件强绑定，未来可以直接接入其他品牌的充电桩，具备开放接入能力。
网关为什么选 MQTT，而不是其他 IoT 协议？
IoT 网关接入云端有多种协议可选，我们对比后选择了 MQTT:
协议特点为什么不选
MQTT发布/订阅模型，协议轻量 (最小 2 字节报文)，支持 QoS 分级，天然支持断线重连和遗嘱消息，IoT 领域最成熟的选择选它
HTTP/REST请求-响应模型，开发者最熟悉每次上报都要建连/断连，开销大；不支持服务端主动推送 (下行指令需要设备轮询)；不适合高频实时数据上报
CoAP轻量级，基于 UDP，适合极低功耗设备基于 UDP 不可靠，NAT 穿透困难；我们的网关是插电设备不需要极致省电；Broker 生态远不如 MQTT 成熟
AMQP企业级消息队列协议，功能丰富协议复杂、报文开销大，ESP32 等嵌入式设备资源不足以支撑；定位偏向服务器间通信，不适合设备端
私有 TCP自定义二进制协议，性能最高需要自己实现连接管理、心跳、重连、消息确认等基础设施；没有现成 Broker，运维成本高
核心理由: MQTT 是 IoT 设备接入云端的事实标准。协议轻量，ESP32 等嵌入式设备有成熟客户端库；发布/订阅模型天然适合"设备上报 + 云端下发"的双向通信场景；QoS 分级让我们可以对不同数据设置不同的可靠性级别；EMQX Broker 经过大规模生产验证，支持百万级连接。Topic 模型非常灵活 — 网关需要上报 CT 电流数据、DLM 状态、充电桩协调信息等自定义数据，随时加字段加 Topic，不受任何标准约束。

几个重要层存在的意义和作用

设备接入层 — 为什么单独成层？
设备协议的复杂性(OCPP状态机、MQTT连接管理、消息编解码)不应该"泄漏"到业务层。接入层的职责就是: 把不同协议的设备消息翻译成统一的内部格式，投递到 Kafka。业务层只消费 Kafka 消息，完全不关心消息来自 OCPP 还是 MQTT。这样做的好处: 未来新增一种接入协议(如 Modbus、自定义TCP)，只需要新增一个接入模块，业务层零改动。
消息层 (Kafka) — 为什么不直接写数据库？
设备消息的特点是: 高频、突发、不可控。1000台设备同时上报数据，如果直接写数据库，连接池会瞬间耗尽。Kafka 在中间起三个作用:
1. 削峰 — 突发流量先进 Kafka 队列，消费端按自己的节奏处理
2. 解耦 — 一条设备消息可以被多个消费者处理(存时序库的、触发告警的、推送APP的)，互不影响
3. 可重放 — 消息在 Kafka 中保留一定时间，如果消费端处理出错，可以重新消费，不会丢数据
业务层 — 核心枢纽
接入层解决"数据怎么进来"，业务层解决"数据进来之后怎么用"。有了业务层，数据才变成用户看得懂的充电记录、用得上的电费账单、收得到的告警通知
六层不是简单的从上到下串联。业务服务层是核心枢纽，向上承接用户请求，向下消费设备消息，两股数据流在这里汇合处理。

这是"设备上云"的最小闭环: 设备能连上来 (protocol + mqtt) → 知道有哪些设备在线 (device) → 有人能登录后台看 (system)。这四个模块跑通了，就意味着第一台充电桩/网关真正"活"在了云端上。之后所有的业务功能都是在这个基础上叠加。

三、关键设计
协议接入、数据保障与 OTA

平台最核心的三个技术设计: 双协议接入、关键数据零丢失保障、远程固件升级。

OCPP通道

充电桩直连: OCPP 1.6-J

充电桩通过 WebSocket 长连接直接与云平台通信，采用 OCPP 1.6-J 协议。OCPP (Open Charge Point Protocol) 是充电桩行业的事实标准，定义了充电桩与中央系统之间的完整交互流程。

OCPP 消息方向用途
BootNotification桩 → 云充电桩启动注册，上报型号、固件版本等
Heartbeat桩 → 云心跳保活，云端判断在线状态
StatusNotification桩 → 云状态变更 (空闲/充电中/故障等)
MeterValues桩 → 云充电计量数据 (电压/电流/功率/电量)
StartTransaction / StopTransaction桩 → 云充电事务开始/结束，携带电量快照
RemoteStartTransaction云 → 桩远程启动充电
RemoteStopTransaction云 → 桩远程停止充电
ChangeConfiguration云 → 桩远程修改充电桩配置
UpdateFirmware云 → 桩触发固件更新

OCPP 消息进入云平台的路径

1
充电桩通过 WebSocket 连接 neuron-protocol 模块 (Netty 服务, 端口 9001)
2
neuron-protocol 解析 OCPP 消息，转换为统一内部格式，投递到 Kafka 对应 Topic
3
业务模块 (neuron-device / neuron-charge / neuron-alert) 从 Kafka 消费，执行业务逻辑
4
下行指令 (远程控制/OTA) 由业务模块投递到 Kafka[command]，neuron-protocol 消费后转为 OCPP 消息发送给充电桩
MQTT通道

网关设备接入: MQTT

网关设备 (N3 Lite / N3) 通过 MQTT 协议接入云端 EMQX Broker。网关是数据汇聚点 — 它不仅上报自身数据，还聚合下挂充电桩的信息一并上报。

MQTT 消息进入云平台的路径

1
网关通过 WiFi 连接 EMQX Broker，MQTT 用户名/密码认证
2
neuron-mqtt 模块订阅 EMQX，接收消息后转换为统一内部格式，投递到 Kafka 对应 Topic
3
业务模块从 Kafka 消费 — 与 OCPP 通道汇合，业务层处理逻辑完全相同

双通道统一: 消息归一化

核心设计: 不论是 OCPP 充电桩的 StartTransaction，还是 MQTT 网关的充电事件，进入 Kafka 后都被转换为同一种内部消息格式。业务层 (neuron-charge) 只处理一种消息结构，完全不知道也不需要知道数据来自哪个协议。

这意味着: 新增一种接入协议 (比如 Modbus TCP)，只需要新增一个接入模块做消息转换，所有业务模块零修改。这就是"接入层与业务层解耦"的真正价值。
数据分级

数据分级: 不是所有数据都需要同等对待

首先要明确: 不同类型的数据，可靠性要求不同。实时功率数据每秒上报一次，丢一两条无所谓；但充电记录如果丢了一条，用户的充电历史就缺了一块，计费就对不上账。所以我们先做数据分级:

数据级别数据类型丢失后果可靠性目标
S级 (零丢失)充电记录 / 计费数据 / 交易结算用户数据缺失，计费对不上账，商业纠纷端到端零丢失，业务级ACK确认
A级 (极低丢失)告警事件 / 设备状态变更 / OTA结果运维漏报告警，设备状态不一致QoS 1 保证送达，消费端幂等处理
B级 (允许少量丢失)实时遥测 (功率/电流/电压)监控图表短暂中断，不影响业务QoS 0 尽力送达，高频弥补
C级 (允许丢失)心跳在线状态延迟更新 (TTL机制兜底)QoS 0，缺失通过超时检测
为什么不全部用最高可靠性？
可靠性有成本。QoS 1 的消息需要 Broker 和客户端来回确认，网络开销翻倍。如果1000台设备每秒上报的遥测数据都用 QoS 1，EMQX 的压力会大幅增加。把有限的可靠性投资用在最值得保护的数据上，这才是合理的工程决策。

S级数据: 端到端零丢失的保障链路

以"充电记录"为例，我们来看一条数据从设备到最终落库，经过了哪些保障环节:

1
设备端本地持久化 — 充电记录先写入设备本地存储 (Flash/文件系统)。即使WiFi断了、云端挂了，数据不会丢。来网后补传。
2
传输层 QoS 1 — MQTT QoS 1 保证消息至少送达一次 (PUBACK确认)。OCPP 基于 WebSocket + CALL/CALLRESULT 机制保证请求-响应匹配。
3
Kafka 持久化 — 消息进入 Kafka 后立即持久化到磁盘，配置 acks=all 确保多副本写入成功。即使某个 Kafka Broker 宕机，数据不丢。
4
消费端幂等处理 — 消费者从 Kafka 取出消息后，先检查 msgId 是否已处理过 (Redis 去重)。相同消息重复投递不会导致重复入库。
5
业务ACK回执 — 数据成功落库后，云端通过下行通道向设备发送业务级确认。设备收到确认后，才从本地存储中删除该条记录。如果超时未收到确认，设备会重传。
6
对账机制 (预留) — 定时对比设备端的记录序号和云端的入库序号，发现缺口则触发补传。这是最后一道防线。
设计思路: 不依赖任何"单一环节"来保证可靠性。每一层都有自己的保障手段:
  设备本地存储 → 防止网络断开时丢数据
  QoS 1 → 防止传输过程中丢数据
  Kafka 持久化 → 防止云端处理故障丢数据
  幂等消费 → 防止重试导致数据重复
  业务ACK → 端到端确认，形成闭环
  对账 → 最终一致性兜底

任何一个环节失败，下一个环节都能兜住。这就是"多层防御"的思路 — 不怕某一层出问题，怕的是所有层同时出问题，而那几乎不可能。

告警数据 (A级) 的保障

告警事件虽然不涉及计费，但漏报可能导致安全隐患被忽视。我们的保障策略:

  • 传输: QoS 1 保证送达
  • Kafka: event Topic 独立，不与遥测混在一起，避免被大量遥测消息"淹没"
  • 消费: 告警消费者独立部署 (未来拆分时可独立扩缩容)，处理失败立即进入死信队列 (DLQ)，人工介入
  • OCPP 端: StatusNotification 错误状态自动触发告警规则
  • 补偿: 遗嘱消息 (MQTT LWT) 和连接断开检测 (OCPP WebSocket) 双重保障离线告警

Kafka Topic 分离策略

Kafka Topic数据来源消费者为什么单独一个Topic
device.telemetryOCPP MeterValues + MQTT 遥测时序存储服务量大、高频，写入 TimescaleDB，不阻塞其他消费
device.eventOCPP StatusNotification + MQTT 事件告警服务、设备状态服务告警需要低延迟处理，不能被遥测队列积压影响
device.billingOCPP Start/StopTransaction + MQTT 计费充电管理服务、计费服务S级数据，独立Topic便于做独立的可靠性保障和监控
device.command业务层下行指令OCPP网关、MQTT网关下行指令需要单独的消费优先级，不与上行混用
为什么不把所有消息放一个 Topic？
混合 Topic 的问题: 一秒钟涌入1000条遥测消息，后面跟着1条告警 — 告警要等遥测都处理完才轮到它，延迟可能达到秒级。分 Topic 让每类消息有独立的消费速度和背压机制。遥测消息积压不影响告警及时处理，计费数据可以配置更高的持久化级别，互不干扰。
离线数据补传策略（MQTT 通道）
网关断网期间数据在本地持续积累，恢复联网后需要处理两个问题：单条消息过大（ESP32 内存有限、Broker 有消息大小上限）和补传冲击实时数据。

补传策略

  • 分批补传，每批 ≤ 20 条，逐批等待云端 ACK 后再发下一批
  • 补传消息标记 "offline": true，云端区分补传和实时数据，分开处理
  • 补传完毕后再恢复实时上报，保证实时数据不被补传挤占

传输优化路径（按阶段演进）

阶段方案效果
阶段 1原始 JSON key开发调试友好，可读性优先
阶段 2压缩 key（"voltage" → "v"）减少 30-40% 体积
阶段 3MessagePack（二进制 JSON）再减 30%，ESP32 有成熟库
阶段 4Protobuf（强 Schema 约束）压缩率最高，适合设备量大之后

OCPP 通道天然是逐条请求-响应模式，不存在大包问题。

OTA设计

OTA 远程升级

OTA 是云平台的核心能力之一。没有 OTA，每次固件更新都需要上门刷机；有了 OTA，产品可以持续迭代，Bug 可以快速修复。

升级流程

1
上传固件: 管理员通过后台上传固件包 → 存入 MinIO → 生成下载URL和校验码
2
创建升级任务: 选择固件版本 + 目标设备 (支持单台/批量/灰度)。灰度策略: 先升10% → 观察24h无异常 → 全量推送
3
下发升级指令: OCPP 通道使用 UpdateFirmware 消息；MQTT 通道发布到 neuron/{gw}/down/ota
4
设备下载固件: 设备通过 HTTP(S) 从 MinIO 直接下载固件包 (不经过应用服务器，避免带宽瓶颈)
5
校验安装: MD5/SHA256 校验 → 写入OTA分区 → 重启。校验失败则丢弃，不冒险安装
6
结果上报: 重启后上报新固件版本 → 云端标记升级成功。失败则自动回滚到上一版本

安全措施

风险措施理由
固件被篡改MD5/SHA256 校验 + 可选固件签名验证确保设备只安装官方固件
下载链接泄露MinIO 临时签名URL (30分钟过期)即使URL泄露也无法长期使用
升级失败变砖双OTA分区 + 自动回滚写入新分区，启动失败自动切回旧分区
批量升级压垮网络灰度策略 + 并发限流避免所有设备同时下载导致服务过载

数据库选型: 为什么选 PostgreSQL

数据库是平台的基石，选型需要审慎。我们对比了市场上主流的关系型数据库，结合充电桩云平台、海外市场、AWS 部署三个核心背景做出选择。

第一轮淘汰: 明确不适合的选项

数据库特点淘汰理由
Oracle企业级商业数据库，功能最全，稳定性极高商业授权费用高昂，我们的数据规模 (万级设备) 完全不需要 Oracle 的能力，用它是杀鸡用牛刀，且增加运维和人力成本
SQL Server微软生态，Windows 平台集成度高与 Java + Linux + Docker 部署方案不搭，AWS 上的支持力度不如 MySQL/PostgreSQL，海外 IoT 领域使用率低
MongoDB文档型数据库，Schema 灵活，适合非结构化数据充电记录、计费账单、用户信息是强结构化数据，需要事务保证和复杂关联查询，关系型数据库更合适。用 MongoDB 管理计费数据存在事务一致性风险
TiDB分布式 NewSQL，兼容 MySQL 协议，水平扩展面向百万级 QPS 和 PB 级数据场景。我们初期几千台设备、每天几万条充电记录，单机 PostgreSQL 绰绰有余。引入分布式数据库会带来不必要的运维复杂度

最终对决: MySQL vs PostgreSQL

淘汰了明显不适合的选项后，剩下 MySQL 和 PostgreSQL — 两者都是优秀的开源关系型数据库，都能满足业务需求。最终选择取决于哪个更契合我们的具体场景。

最终选型: PostgreSQL — 为什么？
在基础能力打平的前提下，PostgreSQL 在我们的核心场景上有明确优势:
  • 多租户更安全 — RLS 在数据库层面强制隔离，未来开放给运营商入驻时，数据安全有底层保障
  • 少一个组件 — TimescaleDB 替代独立时序库，业务数据和时序数据在同一个实例，可以直接 JOIN 查询 (如"过去1小时各设备的平均功率")，运维成本减半
  • 海外市场契合 — OCPP 生态、欧美开发者、AWS 原生支持，与我们的出海战略一致
  • PostGIS 开箱即用 — 充电站地图搜索"附近的充电桩"是必备功能，PG 直接支持，MySQL 需要额外方案
核心逻辑: 基础能力相当时，选更契合业务场景的那个。PostgreSQL 在多租户隔离、时序数据、地理查询、海外生态四个方面的优势，恰好对应了我们平台的四个核心需求。

数据存储策略

PostgreSQL — 业务数据

  • 设备注册信息与拓扑
  • 充电会话记录
  • 用户 / 租户 / 角色
  • 告警记录、OTA任务
  • 计费账单 (未来)

适合: 关系型、事务性、需要复杂查询。RLS 保障多租户隔离。

TimescaleDB (PG 插件) — 时序数据

  • 实时遥测 (电压/电流/功率)
  • 充电功率曲线
  • 设备在线率统计
  • 环境数据 (温度)

适合: 高频写入、时间范围聚合、自动降采样。与业务表可直接 JOIN。

为什么不把所有数据都存普通 PostgreSQL 表？
遥测数据每秒可达数千条写入，普通表的 B-Tree 索引和事务机制会成为写入瓶颈，时间范围聚合查询效率也不理想。TimescaleDB 将时序数据自动按时间分片 (Hypertable)，写入吞吐量提升数十倍，内建时间窗口聚合函数，查询"过去1小时的平均功率"仅需毫秒级响应。它本质上还是 PostgreSQL — 同一个数据库实例、同一个连接、同一套 SQL 语法，只是存储引擎针对时序场景做了优化。

未来: 微服务拆分 (按需)

拆分条件 (什么时候拆)
  • 团队分组独立开发，互相等待、互相阻塞
  • 某个模块需要独立扩缩容 (如设备接入量暴增)
  • 设备量到万台级，单机性能出现瓶颈
  • 需要独立升级某服务而不影响其他
拆分路径 (怎么拆)
  • 每个模块加 @SpringBootApplication + 独立配置
  • 注册到 Nacos 注册中心 + 配置中心
  • @Autowired 改为 @FeignClient 远程调用
  • 引入 Spring Cloud Gateway 统一路由
  • 独立数据库，按服务拆分数据库, 实现数据隔离，

    单体时通过表名前缀划分归属、禁止跨模块外键，保证未来按模块拆库时，每个服务的数据边界是清晰的，拆分成本可控

为什么现在不拆？
初期设备量级和业务复杂度不需要微服务带来的额外运维成本。模块边界在设计阶段就已明确划定 (每个 Maven module = 未来一个独立服务)，这是成本最低的微服务准备。真正需要拆分时，改动范围可控: 增加独立启动类、替换调用方式、增加网关路由。相比一开始就引入微服务架构所带来的分布式事务和服务治理复杂度，先单体后拆分的路径更为高效。

设计原则总结

#原则为什么
1双协议并存，统一数据模型OCPP 拥抱行业标准，MQTT 保持灵活性，业务层不感知协议差异，新增协议零改动
2关键数据零丢失充电记录/计费数据直接关系用户信任和商业结算，多层防御保证端到端可靠
3离线安全，来网即同步家庭WiFi不可靠，设备必须独立工作。云端只做管理增值，不做实时控制
4模块化单体，预留边界当前规模不需要微服务的复杂度，但模块边界保证了未来拆分的低成本
5先闭环再扩展Phase 1 跑通设备上云闭环，验证数据链路，再在此基础上叠加业务
6多租户从第一天开始未来运营商入驻是配置而非代码修改，提前设计比事后改造成本低10倍
7Kafka 分Topic 分级处理不同类型的数据有不同的时效性和可靠性要求，混在一起会互相影响
必须坚持
  • 云端只下发配置，不参与实时控制环路
  • 设备接入层与业务层通过 Kafka 解耦
  • S级数据 (计费/充电记录) 全链路有保障
  • 模块边界 == 未来微服务边界
  • 每一个 OCPP / MQTT 消息都转为统一内部格式
必须避免
  • 业务层直接处理 OCPP/MQTT 原始消息 (耦合)
  • 所有消息放同一个 Kafka Topic (互相阻塞)
  • 过早拆分微服务 (分布式单体陷阱)
  • 设计假设设备永远在线的功能
  • S级数据只依赖单一保障手段
