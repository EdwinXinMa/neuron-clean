POSITIONING

1.1 平台定位

N3 Lite 云平台是一个 B 端管理后台 — 服务于我们的设备运维管理平台。

这个定位决定了前端的几个基本特征：

  • PC 优先 — 用户在办公电脑上操作，优先PC端
  • 数据密集 — 大量表格、列表、搜索、筛选，信息密度高
  • 使用频率适中 — 不是高频交易系统，但要求响应流畅
  • 权限分层 — 不同角色看到不同菜单和数据

1.2 用户角色

角色职责核心操作
管理者 掌握全局 + 系统管理 看仪表盘、看统计、用户管理、角色管理、权限分配、菜单管理
运维人员 日常运维 搜设备、诊断、处理告警、管固件、推 OTA、操作日志

设备台账和设备列表有什么区别？
设备台账是"出厂登记本" — 记录设备的出厂信息（序列号、型号、批次、出给了哪个经销商），在设备出货时批量录入。设备还没通电上线时就已经有台账记录了。

设备列表是"运行时状态" — 展示所有已知设备的当前状态（在线/离线/故障），是运维人员日常查看设备情况的入口。设备通电上线后自动出现在这里。

两者通过序列号关联：设备上线时自动匹配台账记录，如果台账里没有该序列号，系统也允许接入（标记为"未登记设备"），后续可以补录台账。

交互要点

  • 数字卡片可点击 — 点"故障 3"跳转到设备列表（状态=故障）
  • 地图标点可点击 — 跳转到对应设备的诊断页

3.3 客服诊断页

定位：当下怎么了。核心原则：搜索要快，聚焦当前状态，操作要少点击。适用于快速定位问题时使用。

交互流程

1
搜索 — 页面顶部搜索栏，支持按设备序列号搜索。输入即搜，下拉显示匹配结果。
2
选择 — 从下拉结果中选择一台设备，页面立刻展开该设备的当前状态。
3
诊断 — 一屏展示设备当前状态和问题，快速判断。
4
操作 — 如需要，直接在页面内修改 DLM 参数或推送 OTA，不用跳转到其他页面。

诊断页和详情页有什么区别？
诊断页 = 当下怎么了 — 只看现在的状态和问题，一屏看完，快速定位问题时使用。不展示设备基本信息（序列号、型号、批次等），不展示历史记录。

详情页 = 这台设备的一切 — 完整档案 + 历史记录，Tab 页签组织，从设备列表点击进入，适合深入查看和管理。

四、数据流设计
数据从哪来，怎么流转

每个页面需要什么数据，通过什么方式获取。

DATA FLOW

4.1 两种数据通道

前端获取数据只有两个通道：

HTTP 请求（主动拉取）
前端发起 RESTful API 请求，后端返回 JSON 数据。
用于：页面加载、搜索查询、表单提交、文件上传。
绝大多数数据走这个通道。
WebSocket 推送（被动接收）
后端主动推送数据到前端，前端实时更新界面。
仅用于：仪表盘大屏设备在离线数量更新、全局告警角标更新。
使用范围很小，只服务于需要实时感知的场景。

WebSocket 推送事件

#事件类型推送内容前端响应
1设备上线设备序列号、上线时间仪表盘数字卡片更新
2设备离线设备序列号、最后心跳时间仪表盘数字卡片更新
3新告警告警级别、告警类型、设备序列号、描述导航栏角标+1，仪表盘告警动态插入新条目
4告警已处理告警 ID导航栏角标-1

WebSocket 消息格式

{
  "type": "DEVICE_ONLINE",        // 事件类型
  "payload": {                    // 事件数据
    "serialNumber": "NL-20260301-0001",
    "timestamp": 1710000000
  }
}

WebSocket 连接管理

  • 建立时机：用户登录成功后立即建立连接
  • 认证方式：连接时携带 JWT Token
  • 断线重连：断开后自动重连，间隔递增（1s → 2s → 4s → 8s → 最大 30s）
  • 心跳保活：每 30 秒发送 ping，服务端回复 pong
  • 销毁时机：用户登出或关闭页面时断开连接

通用 API/api/common/**文件上传/下载、数据字典

4.4 统一响应格式

所有 HTTP API 返回统一的 JSON 结构：

{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "result": { ... },       // 业务数据
  "timestamp": 1710000000
}

分页查询返回：

{
  "success": true,
  "code": 200,
  "result": {
    "records": [ ... ],    // 当前页数据
    "total": 128,          // 总条数
    "current": 1,          // 当前页码
    "size": 10             // 每页条数
  }
}

错误处理策略

HTTP 状态码含义前端处理
200成功（看 success 字段判断业务是否成功）正常处理
401未认证 / Token 过期跳转登录页
403无权限提示"无权限访问"
500服务器错误提示"服务异常，请稍后重试"

REQUIREMENTS

5.1 业务需求对前端的技术要求

#业务需求技术要求优先级

5.2 需要具备的基础能力

除了直接对应业务需求的能力外，一个 B 端管理后台还需要以下基础设施。如果选型时能找到已经内置这些能力的成熟模板/脚手架，可以大幅减少基础设施开发量，让团队聚焦业务页面。

必须具备
  • 登录认证 — JWT Token 管理、自动刷新、过期跳转
  • 权限体系 — 路由守卫（无权限页面不可访问）+ 按钮权限（无权限按钮不显示）
  • 动态菜单 — 菜单配置从后端加载，支持角色级菜单隔离
  • HTTP 请求封装 — 统一拦截器（Token 注入、错误处理、Loading 状态）
  • 布局框架 — 左侧菜单 + 顶部导航 + 多 Tab 页签 + 内容区的标准管理后台布局
  • CRUD 封装 — 可配置化的表格 + 搜索表单 + 弹窗表单，避免每个页面重复写模板代码
应该具备
  • 主题切换 — 亮色/暗色模式，满足不同使用环境偏好
  • 数据字典 — 下拉选项、状态标签等从后端字典表加载，无需前端硬编码
  • 文件处理 — 统一的上传/下载封装，支持进度展示
  • 消息通知 — 全局消息提示、操作确认弹窗的统一交互
  • 构建优化 — 路由懒加载、组件按需导入、生产环境压缩
  • 代码规范 — 统一的格式化规则 + 提交前自动检查

STRUCTURE

6.1 目录划分原则

无论最终选择什么技术栈，前端工程的目录都应遵循以下划分原则：

目录职责存放内容划分规则
API 层后端接口定义 + 请求/响应类型按后端模块分目录（设备、告警、系统），每个模块一个接口文件 + 一个类型定义文件
页面层业务页面组件按菜单结构分目录，与导航结构（第二章）一一对应
组件层可复用的公共组件只有被 2 个以上页面复用的组件才放公共目录；只在一个页面内使用的子组件放在该页面的子目录
状态层全局共享状态只有跨页面共享的状态才放全局（如用户信息、告警通知）；页面内部状态留在组件本地
工具层工具函数、常量枚举通用工具（日期格式化、文件处理等）+ 业务枚举（设备状态、告警级别）

页面目录与菜单的对应关系

页面目录/
├── dashboard/              ← 首页仪表盘
│   ├── 主页面文件
│   └── components/         ← 仪表盘子组件（数字卡片、地图、告警动态、固件分布）
├── device/                 ← 设备管理
│   ├── list/               ← 设备列表
│   ├── detail/             ← 设备详情（Tab 页签结构）
│   └── ledger/             ← 设备台账
├── operation/              ← 运维中心
│   ├── diagnosis/          ← 客服诊断页
│   │   └── components/     ← 搜索组件、状态快照、DLM 面板、拓扑树
│   ├── alert/              ← 告警监控
│   └── oplog/              ← 操作日志
├── firmware/               ← 固件升级
│   ├── list/               ← 固件管理
│   └── task/               ← OTA 升级任务
└── system/                 ← 系统管理（基座自带）
为什么页面目录要和菜单结构对齐？
开发者看到菜单上有"运维中心 → 客服诊断"，就知道代码在 operation/diagnosis/ 目录下，不用猜、不用搜。菜单结构 = 目录结构 = 路由结构，三者统一，维护成本最低。

6.2 页面开发模式

9 个业务页面按类型可分为两种开发模式：

模式一：配置驱动的 CRUD 页面
适用：设备台账、设备列表、告警列表、固件管理、操作日志
核心思路：表格列定义和搜索条件写成配置，页面组件套用统一模板。

开发一个 CRUD 页面只需三步：
1. 定义数据配置（表格有哪些列、搜索有哪些条件）
2. 编写页面组件（套用 CRUD 模板，传入配置）
3. 定义 API 接口（对接后端 CRUD 接口）

一个标准 CRUD 页面预计 30 分钟完成。
模式二：定制布局的特殊页面
适用：首页仪表盘、客服诊断页、设备详情页、OTA 升级任务
核心思路：拆成多个子组件，各自独立获取数据，组合到一个页面布局中。

以客服诊断页为例：
1. 搜索组件 — 负责设备搜索
2. 状态快照组件 — 展示当前状态和问题
3. DLM 操作面板 — 快捷操作
4. 拓扑树组件 — 展示父子关系
5. 主页面 — 组合以上子组件，控制整体布局

每个子组件职责单一，独立可测试。

6.3 开发规范

必须遵守
  • API 请求必须经过统一的接口层，页面组件中禁止直接写请求 URL
  • API 接口的请求参数和返回值必须定义类型，禁止使用 any
  • 页面入口文件统一命名，子组件放在页面子目录中
  • 跨页面复用的组件放公共组件目录，单页面使用的组件留在页面内
  • 枚举值使用常量定义，禁止魔法数字和魔法字符串
禁止事项
  • 禁止跨模块直接引用其他模块的内部子组件
  • 禁止在全局状态中存放只有一个页面使用的临时数据
  • 禁止绕过权限体系直接控制页面/按钮的显示隐藏
  • 禁止在业务代码中直接操作 DOM，应通过框架的响应式机制更新界面
