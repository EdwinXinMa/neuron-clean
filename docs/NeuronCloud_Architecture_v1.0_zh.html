NeuronCloud 智能充电云平台

架构设计规格说明书 — 旭衡电子(深圳) AlwaysControl Technology

概要总览

核心决策
  • 架构模式: 模块化单体，预留微服务拆分边界
  • IoT协议: MQTT over WiFi (主通道)，私有协议 (预留)
  • 消息管道: EMQX Broker + Kafka (缓冲/解耦/可重放)
  • 数据策略: MySQL (业务) + TDengine (时序) + MinIO (对象)
  • 设备拓扑: 网关-子设备模型 (N3 Lite = 网关, ATP III = 子设备)
交付策略
  • Phase 1: 设备上线 + OTA + 基础监控 (N3 Lite 优先)
  • Phase 2: 充电管理 + 能耗分析 + 告警 + APP
  • Phase 3: 计费结算 + 运营商管理 + 开放API
  • Phase 4: 光储充一体化, V2G, 数据分析
  • 原则: Phase 1 先跑通闭环，验证后再扩展
为什么做这个架构

N3 Lite 目前是一台离线设备。没有云平台，每次固件Bug都需要上门刷机，用户看不到充电数据，AlwaysControl对设备运行状态一无所知。 云平台将一次性硬件销售转变为持续的服务关系 — 远程诊断、OTA升级、使用分析，最终实现可盈利的充电服务。硬件是入口，云平台是护城河。

术语表

DLM (动态负载管理)
运行在 N3/N3 Lite 上的核心算法。CT环测量家庭总电流，DLM 动态调节充电功率，防止空开跳闸同时最大化充电速度。
CT环 (电流互感器)
安装在入户主线上的钳式传感器，为 N3/N3 Lite 提供实时电流测量数据用于 DLM 计算。N3 Lite 支持单相(100A)；N3 支持三相。
PLC (电力线通信)
N3/N3 Lite 与 ATP III 充电桩之间的通信方式，复用现有充电电源线。P1901.1 标准，OFDM+FSK 调制，无需额外布线。
OCPP (开放充电协议)
充电桩管理的行业标准协议 (1.6-J / 2.0.1 版本)。用于充电桩直连云平台的备选通道。
网关-子设备模型
N3/N3 Lite 作为 MQTT 网关（单一云端连接），ATP III 充电桩作为子设备，数据通过网关聚合上报。
CP信号 (控制导引)
IEC 61851 Mode 3 充电控制信号。PWM 占空比编码允许的充电电流 (6A-32A)。ATP III 根据 N3/N3 Lite 发出的 CP 信号调整输出。
EVSE (电动汽车供电设备)
为电动车供电的充电站硬件。ATP III 即为 EVSE，配备 AC Type 2 接口，支持 7.7kW 至 23kW 输出。
OTA (空中升级)
远程固件升级机制。N3 Lite (ESP32 双分区) 和 ATP III 均支持独立 OTA。N3 Lite 通过 PLC 中转 ATP III 的固件。

第三章
MQTT协议设计

设备通信协议: Topic命名、消息格式、QoS策略与安全机制。

协议规格

MQTT通信设计

认证方式

MQTT 用户名/密码认证。用户名 = 设备SN, 密码 = 设备专属密钥。EMQX ACL 规则实现按设备的 Topic 访问控制。

QoS策略

数据类型QoS等级理由
遥测数据 (实时功率)QoS 0高频上报, 偶尔丢失可接受
心跳QoS 0周期性上报, 丢失通过TTL过期检测
事件 / 告警 / 状态QoS 1需保证至少送达一次
充电记录QoS 1 + 业务ACK计费数据零丢失
下行指令QoS 1确保送达设备

Topic规格

方向TopicQoS说明
上行neuron/{gw}/up/telemetry0实时电压/电流/功率/CT数据/子设备状态
neuron/{gw}/up/heartbeat0在线保活 (分钟级间隔)
neuron/{gw}/up/event1告警、状态变更、充电启停事件
neuron/{gw}/up/billing1充电会话记录、电量快照
neuron/{gw}/up/reply1指令执行结果 (响应下行指令)
下行neuron/{gw}/down/command1启/停充电、设电流、DLM配置、断路器档位、优先级
neuron/{gw}/down/config1DLM参数、充电模式、预约计划
neuron/{gw}/down/ota1OTA固件推送 (目标: n3lite / atp3_1 / atp3_2)

遗嘱消息 (LWT)

每台 N3 Lite 连接时注册一条遗嘱消息到 neuron/{gw}/up/event，负载为 {"type":"offline"}。设备意外断连时，EMQX 自动发布此消息，触发云端离线告警流程。

OTA设计

OTA远程升级

N3 Lite (ESP32) 原生支持双分区OTA。云平台提供固件管理与升级编排。ATP III 固件通过 N3 Lite 经 PLC 中转。

OTA流程

1. 上传固件 管理员上传 .bin 至云端 -> 存入 MinIO -> 返回 URL + MD5
|
2. 创建升级任务 选择固件版本 + 目标设备 (单台 / 批量 / 灰度)
|
3. MQTT推送OTA指令 云端发布到 neuron/{gw}/down/ota (QoS 1)
|
4. 设备下载固件 N3 Lite 通过 HTTP(S) 从 MinIO 获取 .bin
|
5. 校验与安装 MD5 校验 -> 写入OTA分区 -> 重启
|
6. 上报结果 重启后心跳携带新固件版本 -> 云端标记升级成功
ATP III OTA: N3 Lite 下载固件后通过 PLC 中转至指定充电桩。OTA指令中通过 "target": "atp3_1" 指定目标设备。

OTA指令消息

{
  "msgId": "ota-uuid-001",
  "action": "upgrade",
  "data": {

安全措施

方面措施
传输安全MinIO 内网访问, 或 HTTPS + 临时签名URL (30分钟过期)

回滚机制ESP32 双OTA分区; 启动失败自动回滚

并发控制批量升级限流, 防止 MinIO/网络过载

单体到微服务演进

当前: 模块化单体
  • 所有模块共享一个JVM进程, 一个数据库连接池
  • 模块边界通过 Maven 模块结构强制
  • 模块间通过 Spring Bean 注入调用
  • 部署简单, 单个 Docker 容器
未来: 微服务 (按需拆分)
  • 每个模块加 @SpringBootApplication + 独立配置
  • 注册到 Nacos (项目中已有配置)
  • @Autowired 替换为 @FeignClient 跨模块调用
  • 引入 Spring Cloud Gateway 统一API路由
何时拆分: 团队 <= 5人、设备 <= 数千台、单机性能足够时保持单体。当团队互相阻塞、某模块需要独立扩缩容(如MQTT接入高峰)、或设备数达到万台以上时考虑拆分。

实施阶段

Phase 1 — 基础建设 当前
  • 项目框架 (Spring Boot 3.x, neuron-* 模块)
  • MQTT + Kafka 消息通道 (neuron-mqtt)
  • 设备管理 + OTA (neuron-device)

Phase 2 — 核心业务 下一步
  • 充电管理 (neuron-charge)
  • 能耗分析 (neuron-energy)
  • 告警中心 (neuron-alert)
  • 用户APP (基础版)
  • 管理后台页面 (设备/充电/能耗)

交付物: 用户可通过APP监控和控制充电。

Phase 3 — 商业化 后续
  • 计费模块 (neuron-billing)
  • 运营商管理 (neuron-operator)

交付物: 平台可从充电服务中获得收入。

Phase 4 — 差异化 远期

指导准则

设计原则

#原则说明
1以设备为中心每个功能都从 N3 Lite 实际硬件能力出发。设备不支持的，不做。
2离线优先N3 Lite 的 DLM 和本地充电必须在无云端的情况下正常工作。连接恢复后数据同步。
3数据可靠计费相关数据: 零丢失。QoS 1 + 本地持久化 + 业务ACK + 对账(预留设计)。
4水平扩展MQTT/Kafka 架构支持 100 到 10万台设备。单体可演进为微服务。
5多租户就绪从第一天起考虑租户隔离。未来运营商入驻是配置，不是代码修改。
6协议共存MQTT (N3网关) + OCPP (充电桩直连) 双通道，内部统一数据模型。
7单体优先清晰的模块边界，开发效率优先。只在业务增长需要时才拆分微服务。
必须坚持
  • DLM 算法运行在设备端, 不在云端 (延迟 + 离线安全)
  • 云端只下发配置, 绝不参与实时控制环路
  • 每台网关一个 MQTT 连接 (不是每台充电桩)
  • 模块边界 == 未来微服务边界
必须避免
  • 设计假设设备永远在线的功能
  • 过早拆分微服务 (分布式单体陷阱)
  • 在同一个 Kafka Topic 中混合不同类型的 MQTT 消息
  • 设备协议层与业务逻辑紧耦合
