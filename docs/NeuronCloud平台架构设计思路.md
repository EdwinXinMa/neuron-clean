## 一、产品背景: N3 Lite 是什么

### 1.1 一句话定义

N3 Lite 是一台面向家庭用户的 **智能充电网关**，安装在配电箱内，通过电力线(PLC)控制充电桩，通过 WiFi 连接云平台，核心解决 **"家庭充电不跳闸"** 的问题。

### 1.2 硬件能力

| 项目 | 参数 |
|------|------|
| 芯片 | ESP32 双核 MCU |
| 通信 | WiFi 802.11 / BLE 蓝牙 / PLC 电力线通信 |
| CT环 | 1路(单相), 实时监测入户电流 |
| 管理能力 | ≤ 2 台 ATP III 充电桩 |
| 配网方式 | BLE 蓝牙配网(手机App引导) |
| 上云协议 | MQTT over WiFi |

### 1.3 核心能力

| 能力 | 说明 |
|------|------|
| **防跳闸** | CT环实时监测入户总电流, 家庭负载高时自动降低充电功率, 避免空开跳闸 |
| **动态负载均衡(DLM)** | 2台充电桩智能分配可用容量, 优先级/均衡/排队多种策略 |
| **充电模式** | 即插即充 / 预约充电(利用低谷电价) |
| **电流控制** | 6A - 32A 动态调节, 适配不同家庭电网容量 |
| **电能监测** | 实时功率、累计电量统计 |

### 1.4 产品体系定位

```
┌─────────────────────────────────────────────────────────────┐
│                    AlwaysControl 产品线                      │
├──────────────┬──────────────────┬───────────────────────────┤
│   ATP III    │     N3 Lite      │          N3               │
│   充电桩      │   家庭网关         │       商用网关             │
│   (执行层)    │   (家庭控制层)     │      (商业控制层)           │
├──────────────┼──────────────────┼───────────────────────────┤
│ 6A-32A充电    │ ≤2桩, 单相        │ 多桩, 三相, RS485/CAN      │
│ PLC从站       │ WiFi/BLE/PLC     │ WiFi/4G/ETH/PLC          │
│ 接受指令执行   │ 家庭DLM           │ 商业DLM + 物模型 + 场景联动  │
└──────────────┴──────────────────┴───────────────────────────┘
                         ↑ 全部通过 MQTT 接入
                  ┌──────────────────┐
                  │   NeuronCloud    │
                  │   智能充电云平台    │
                  └──────────────────┘

## 二、为什么要做这个云平台

### 2.1 没有云平台的现状

- N3 Lite 是一个**离线设备**: 本地DLM可以工作, 但用户无法远程查看、无法远程控制
- 设备出了问题, 售后团队无法远程诊断, 必须上门
- 充电数据没有沉淀, 用户不知道每月充了多少度电、花了多少钱
- 每台设备都是孤岛, AlwaysControl 对设备运行状态一无所知
- 无法进行OTA远程升级, 固件有问题必须上门刷机

### 2.2 有了云平台之后

| 角色 | 获得的价值 |
|------|-----------|
| **家庭用户** | 手机App远程监控、查看充电记录、预约充电、电费统计、异常通知 |
| **AlwaysControl(我们)** | 设备全生命周期管理、远程诊断、OTA升级、运营数据大盘 |
| **商业客户(未来)** | 运营后台、计费结算、多站点管理、开放API对接 |

### 2.3 商业模式转变

```
现在:  卖硬件(一次性收入) → 交付即结束
未来:  硬件 + 云服务(持续收入) → 长期价值

硬件是入口, 云平台是护城河。
设备卖得越多, 平台价值越大, 用户粘性越强。
```

```

### 4.2 数据流全景

```
           N3 Lite (家庭现场)                    NeuronCloud (云端)
        ┌─────────────────┐
        │  CT环 → DLM算法  │
        │  PLC ↔ 充电桩    │
        └────────┬────────┘
                 │ WiFi / MQTT
                 ↓
上行数据流:
  遥测数据(实时电流/电压/功率) ──→ EMQX ──→ Kafka[telemetry] ──→ TDengine(时序存储)
  事件告警(过流/离线/故障)     ──→ EMQX ──→ Kafka[event]     ──→ MySQL(告警记录) + 通知推送
  计费数据(充电记录/电量)     ──→ EMQX ──→ Kafka[billing]   ──→ MySQL(计费结算)
  心跳                       ──→ EMQX ──→ Redis TTL        ──→ 在线状态管理

下行数据流:
  远程控制(启停充电/调电流)   ←── EMQX ←── Kafka[command]   ←── 用户App/管理后台
  配置下发(DLM参数/电价)     ←── EMQX ←── 配置服务          ←── 管理后台
  OTA升级(固件URL+MD5)      ←── EMQX ←── OTA服务           ←── 管理后台
  固件下载                   ←── MinIO(HTTP直接下载)         ←── N3 Lite
```

## 七、N3 Lite 上云协议设计

### 7.1 通信协议

```
N3 Lite ←── MQTT over WiFi ──→ EMQX Broker ──→ NeuronCloud

认证: MQTT Username/Password (设备ID + 密钥)
QoS策略:
  - 遥测数据: QoS 0 (允许少量丢失, 高频)
  - 事件/告警: QoS 1 (至少一次送达)
  - 计费数据: QoS 1 + 业务层ACK (不允许丢失)
  - 下行指令: QoS 1 (确保送达)
```

### 7.2 Topic 规范

```
上行(设备 → 云端):
  neuron/{gatewayId}/up/telemetry      实时遥测(电流/电压/功率/温度)
  neuron/{gatewayId}/up/event          事件上报(告警/状态变化/充电事件)
  neuron/{gatewayId}/up/billing        计费数据(充电记录/电量快照)
  neuron/{gatewayId}/up/heartbeat      心跳(在线保活)
  neuron/{gatewayId}/up/reply          指令响应(对下行指令的执行结果)

下行(云端 → 设备):
  neuron/{gatewayId}/down/command      控制指令(启停充电/调节电流)
  neuron/{gatewayId}/down/config       配置下发(DLM参数/充电模式)
  neuron/{gatewayId}/down/ota          OTA升级(固件URL/版本号)
```

## 八、OTA 远程升级设计

N3 Lite 基于 ESP32, 原生支持 OTA 分区升级。云平台提供固件管理和升级调度能力。

### 8.1 OTA 升级流程

```
管理后台                  云平台                    MinIO             N3 Lite
  │                        │                        │                  │
  │  1.上传固件包           │                        │                  │
  │ ─────────────────────→ │  2.存储到MinIO          │                  │
  │                        │ ─────────────────────→  │                  │
  │                        │  ←── 返回URL + MD5      │                  │
  │                        │                        │                  │
  │  3.创建升级任务         │                        │                  │
  │    (选版本+选设备)      │                        │                  │
  │ ─────────────────────→ │                        │                  │
  │                        │  4.MQTT下发OTA指令      │                  │
  │                        │  (QoS 1)               │                  │
  │                        │ ─────────────────────────────────────────→ │
  │                        │                        │                  │
  │                        │                        │  5.HTTP下载固件   │
  │                        │                        │ ←───────────────  │
  │                        │                        │ ────────────────→ │
  │                        │                        │                  │
  │                        │  6.上报进度(downloading/verifying)         │
  │                        │ ←─────────────────────────────────────────│
  │                        │                        │                  │
  │                        │  7.校验MD5 → 写入OTA分区 → 重启            │
  │                        │                        │                  │
  │                        │  8.重启后心跳(携带新版本号)                  │
  │                        │ ←─────────────────────────────────────────│
  │                        │                        │                  │
  │  9.升级结果             │                        │                  │
  │ ←───────────────────── │                        │                  │
```

### 8.2 OTA MQTT 消息定义

```json
// 下行指令: neuron/{gatewayId}/down/ota
{
  "msgId": "ota-uuid-001",
  "action": "upgrade",
  "data": {
    "version": "1.2.0",
    "url": "http://minio:9000/firmware/n3lite/n3lite-v1.2.0.bin",
    "md5": "a1b2c3d4e5f6...",
    "size": 1048576,
    "releaseNote": "修复DLM算法精度问题",
    "forceUpgrade": false
  }
}

// 上行进度: neuron/{gatewayId}/up/reply
{
  "msgId": "ota-uuid-001",
  "type": "ota_progress",
  "data": {
    "status": "downloading",
    "progress": 65,
    "version": "1.2.0",
    "error": null
  }
}
// status: downloading → verifying → installing → rebooting → success / failed
```

### 8.3 云平台 OTA 管理功能

```
固件管理
├── 固件上传(存MinIO, 记录版本/MD5/大小/更新说明/适用机型)
├── 版本列表(版本号/上传时间/状态: 测试版/正式版/已废弃)
└── 固件对比(版本间差异说明)

升级任务
├── 创建任务(选固件版本 + 选目标设备, 支持批量)
├── 灰度策略(先升10% → 观察24h → 无异常 → 全量推送)
├── 任务看板(总数/成功/失败/进行中/待升级)
├── 失败重试(单台/批量)
└── 强制回滚(下发旧版本固件)

升级记录
├── 每台设备的升级历史(时间/版本/结果/耗时)
├── 当前版本分布统计(饼图: 各版本设备占比)
└── 失败原因分析(网络超时/校验失败/空间不足)
```

### 8.4 安全保障

| 环节 | 措施 |
|------|------|
| 传输 | MinIO 内网访问, 或 HTTPS + 临时签名URL(有效期30分钟) |
| 完整性 | MD5/SHA256 校验, 不匹配则丢弃 |
| 回滚 | ESP32双OTA分区, 升级失败自动回滚到上一版本 |
| 防刷 | 固件签名验证(可选, 防止刷入非官方固件) |
| 并发 | 批量升级限流, 避免同时下载导致MinIO/网络压力过大 |

## 十、微服务演进策略

### 10.1 设计原则: 单体先行, 架构预留

初期采用**单体部署**, 但模块按微服务边界划分。每个 Maven module 拥有独立的 Controller/Service/Mapper, 模块间通过 Spring Bean 调用。当业务规模增长到需要拆分时, 可低成本演进为微服务。

### 10.2 当前架构(单体)

```
neuron-system-start (单一JVM进程)
├── neuron-system       系统管理
├── neuron-mqtt          MQTT消息接入
├── neuron-protocol      OCPP协议网关
├── neuron-device        设备管理
├── neuron-charge        充电业务
├── neuron-energy        能源统计
└── neuron-alert         告警中心
```

所有模块共享一个数据库连接池、一个进程, 部署运维简单。

### 10.3 未来架构(微服务)

```
                    ┌──────────────┐
                    │ API Gateway  │  Spring Cloud Gateway
                    │  (网关路由)   │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
     ┌────────────┐ ┌────────────┐ ┌────────────┐
     │ neuron-    │ │ neuron-    │ │ neuron-    │
     │ system     │ │ device     │ │ charge     │  ... 各业务服务
     │ (系统管理)  │ │ (设备管理)  │ │ (充电业务)  │
     └────────────┘ └────────────┘ └────────────┘
              ↑            ↑            ↑
              └────────────┼────────────┘
                    ┌──────┴───────┐
                    │    Nacos     │  注册中心 + 配置中心
                    └──────────────┘

独立部署:
  neuron-mqtt       → 消息接入服务(可水平扩展, 应对大量设备连接)
  neuron-protocol   → OCPP协议服务(独立端口, 独立扩缩容)
```

### 10.4 拆分路径

| 步骤 | 改动内容 | 工作量 |
|------|---------|--------|
| 1. 独立启动类 | 每个模块加 `@SpringBootApplication` + 独立配置 | 小 |
| 2. 服务注册 | 引入 Nacos (项目已有配置基础) | 小 |
| 3. 模块间通信 | `@Autowired` 改为 `@FeignClient` 远程调用 | 中 |
| 4. API网关 | Spring Cloud Gateway 统一入口, 路由分发 | 小 |
| 5. 独立数据库(可选) | 按服务拆分数据库, 实现数据隔离 | 大 |

### 10.5 拆分时机判断

```
保持单体:                         考虑拆分:
  - 团队 ≤ 5人                     - 团队分组独立开发, 互相阻塞
  - 设备 ≤ 数千台                   - 某模块需独立扩缩容(如MQTT接入量暴增)
  - 单机性能满足需求                 - 设备量到万台级, 单体瓶颈
  - 部署运维简单优先                 - 需要独立升级某服务不影响其他
```

> 核心思路: **不为拆而拆**。单体阶段把模块边界、接口契约定义清楚, 就是最好的微服务准备。

## 十一、核心设计原则

1. **以设备为中心**: 一切从 N3 Lite 的实际能力出发, 不做设备做不到的功能
2. **离线优先**: N3 Lite 断网时本地DLM和充电不受影响, 来网后数据补传
3. **数据可靠**: 计费相关数据零丢失(QoS1 + 本地持久化 + 业务ACK + 对账, 预留设计)
4. **水平扩展**: MQTT/Kafka架构, 支撑从100台到10万台设备, 单体可演进为微服务
5. **多租户就绪**: 从第一天就考虑租户隔离, 未来直接支持运营商入驻
6. **协议并存**: MQTT(N3网关) + OCPP(直连充电桩) 双通道, 统一数据模型
7. **单体先行**: 模块边界清晰, 开发效率优先, 规模增长时低成本拆分为微服务
