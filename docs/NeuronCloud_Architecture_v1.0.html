Architecture Design Specification — AlwaysControl Technology

Executive Summary

Core Decisions
  • Architecture: Modular Monolith, microservice-ready boundaries
  • IoT Protocol: MQTT over WiFi (primary), private protocol (reserved)
  • Message Pipeline: EMQX Broker + Kafka (buffer/decouple/replay)
  • Data Strategy: MySQL (business) + TDengine (time-series) + MinIO (objects)
  • Device Topology: Gateway-SubDevice model (N3 Lite = gateway, ATP III = sub-device)
Delivery Strategy
  • Phase 1: Device online + OTA + basic monitoring (N3 Lite first)
  • Phase 2: Charging management + energy analytics + alerts + APP
  • Phase 3: Billing + operator management + open API
  • Phase 4: Solar+Storage+Charging integration, V2G, analytics
  • Principle: Ship Phase 1 first, prove the loop, then expand
Why This Architecture

N3 Lite is an offline-capable device today. Without cloud connectivity, every firmware bug requires on-site service, users have no visibility into charging data, and AlwaysControl has zero insight into fleet health. The cloud platform transforms a one-time hardware sale into a continuous service relationship — remote diagnostics, OTA updates, usage analytics, and eventually monetizable charging services. Hardware is the entry point; the cloud platform is the moat.

Appendix

Glossary

DLM (Dynamic Load Management)
Core algorithm running on N3/N3 Lite. CT ring measures household current; DLM dynamically adjusts charging power to prevent breaker trips while maximizing charge speed.
CT Ring (Current Transformer)
Clip-on sensor installed on the main power line. Provides real-time current measurement to N3/N3 Lite for DLM calculations. N3 Lite supports 1-phase (100A); N3 supports 3-phase.
PLC (Power Line Communication)
Communication between N3/N3 Lite and ATP III chargers using the existing power cable. P1901.1 standard, OFDM+FSK modulation. No additional wiring needed.
OCPP (Open Charge Point Protocol)
Industry standard protocol for EV charger management (versions 1.6-J and 2.0.1). Used for direct charger-to-cloud connections as an alternative channel.
Gateway-SubDevice Model
N3/N3 Lite acts as the MQTT gateway (single cloud connection). ATP III chargers are sub-devices whose data is aggregated and relayed through the gateway.
CP Signal (Control Pilot)
IEC 61851 Mode 3 charging control signal. PWM duty cycle encodes the allowed charging current (6A-32A). ATP III adjusts output based on CP signal from N3/N3 Lite.
EVSE (EV Supply Equipment)
The charging station hardware that supplies power to EVs. ATP III is an EVSE with AC Type 2 connector, supporting 7.7kW to 23kW output.
OTA (Over-The-Air Update)
Remote firmware upgrade mechanism. Both N3 Lite (ESP32 dual-partition) and ATP III support independent OTA. N3 Lite relays ATP III firmware via PLC.

MQTT Protocol Design

Device communication protocol: topic naming, message format, QoS strategy, and security.

Protocol Specification

MQTT Communication Design

Authentication

MQTT Username/Password authentication. Username = Device SN, Password = device-specific secret key. EMQX ACL rules enforce per-device topic access.

QoS Strategy

Data TypeQoS LevelRationale
Telemetry (real-time power)QoS 0High frequency, tolerate occasional loss
HeartbeatQoS 0Periodic, loss is detected by TTL expiry
Events / Alerts / StatusQoS 1At-least-once delivery required
Charging RecordsQoS 1 + Business ACKZero loss for billing data
Downlink CommandsQoS 1Ensure delivery to device

Topic Specification

DirectionTopicQoSDescription
Uplinkneuron/{gw}/up/telemetry0Real-time voltage / current / power / CT data / sub-device status
neuron/{gw}/up/heartbeat0Online keepalive (minute-level interval)
neuron/{gw}/up/event1Alerts, status changes, charging start/stop events
neuron/{gw}/up/billing1Charging session records, energy snapshots
neuron/{gw}/up/reply1Command execution result (response to downlink)
Downlinkneuron/{gw}/down/command1Start/stop charging, set current, DLM config, breaker rating, priority
neuron/{gw}/down/config1DLM parameters, charging mode, schedule
neuron/{gw}/down/ota1OTA firmware push (target: n3lite / atp3_1 / atp3_2)

{gw} = Gateway device SN (N3 Lite or N3). ATP III data is nested as sub-device payloads.

LWT (Last Will and Testament)

Each N3 Lite registers an LWT message on neuron/{gw}/up/event with payload {"type":"offline"}. When the device disconnects unexpectedly, EMQX publishes this message automatically, triggering the cloud-side offline alert workflow.

OTA Design

OTA Remote Upgrade

N3 Lite (ESP32) natively supports dual-partition OTA. The cloud platform provides firmware management and upgrade orchestration. ATP III firmware is relayed through N3 Lite via PLC.

OTA Flow

1. Upload Firmware Admin uploads .bin to cloud -> stored in MinIO -> returns URL + MD5
|
2. Create Upgrade Task Select firmware version + target devices (single / batch / grayscale)
|
3. MQTT Push OTA Command Cloud publishes to neuron/{gw}/down/ota (QoS 1)
|
4. Device Downloads Firmware N3 Lite fetches .bin from MinIO via HTTP(S)
|
5. Verify & Install MD5 check -> write to OTA partition -> reboot
|
6. Report Result Post-reboot heartbeat carries new firmware version -> cloud marks upgrade success
ATP III OTA: N3 Lite downloads firmware, then relays to specific charger via PLC. Target device specified by "target": "atp3_1" in OTA command.

OTA Command Message

// Downlink: neuron/{gw}/down/ota
{
  "msgId": "ota-uuid-001",
  "action": "upgrade",
  "data": {
    "target": "n3lite",           // "n3lite" | "atp3_1" | "atp3_2"
    "version": "1.2.0",
    "url": "https://minio.xxx/firmware/n3lite-v1.2.0.bin",
    "md5": "a1b2c3d4e5f6...",
    "size": 1048576,
    "releaseNote": "Fix DLM precision issue",
    "forceUpgrade": false
  }
}

// Uplink progress: neuron/{gw}/up/reply
{
  "msgId": "ota-uuid-001",
  "type": "ota_progress",
  "data": {
    "target": "n3lite",
    "status": "downloading",      // downloading -> verifying -> installing -> rebooting -> success | failed
    "progress": 65,
    "version": "1.2.0",
    "error": null
  }
}

Monolith-to-Microservice Evolution

Current: Modular Monolith
  • All modules share one JVM process, one DB connection pool
  • Module boundaries enforced by Maven module structure
  • Inter-module calls via Spring Bean injection
  • Simple deployment, single Docker container
Future: Microservice (When Needed)
  • Add @SpringBootApplication per module + standalone config
  • Register with Nacos (config already exists in project)
  • Replace @Autowired with @FeignClient for cross-module calls
  • Add Spring Cloud Gateway for unified API routing
When to split: Stay monolith when team <= 5, devices <= thousands, single-machine performance is sufficient. Consider splitting when teams block each other, a specific module needs independent scaling (e.g., MQTT access spikes), or device count reaches 10K+.

Guiding Rules

Design Principles

#PrincipleDescription
1Device-CentricEvery feature starts from N3 Lite's actual hardware capabilities. Don't build what the device can't support.
2Offline-FirstN3 Lite's DLM and local charging must work without cloud. Data syncs when connectivity resumes.
3Data ReliabilityBilling-related data: zero loss. QoS 1 + local persistence + business ACK + reconciliation (reserved design).
4Horizontal ScalabilityMQTT/Kafka architecture supports 100 to 100K devices. Monolith can evolve to microservices.
5Multi-Tenant ReadyTenant isolation considered from day one. Future operator onboarding is configuration, not code change.
6Protocol CoexistenceMQTT (N3 gateways) + OCPP (direct chargers) dual channels, unified data model internally.
7Monolith FirstClear module boundaries, dev efficiency first. Split to microservices only when growth demands it.
Must Maintain
  • DLM algorithm runs on device, NOT on cloud (latency + offline safety)
  • Cloud only sends configuration, never real-time control loop signals
  • One MQTT connection per gateway (not per charger)
  • Module boundaries == future microservice boundaries
Must Avoid
  • Building features that assume always-online connectivity
  • Splitting into microservices prematurely (distributed monolith trap)
  • Mixing MQTT message types in a single Kafka topic
  • Tight coupling between device protocol layer and business logic
