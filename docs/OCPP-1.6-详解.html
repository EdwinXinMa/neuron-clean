CHAPTER 01

OCPP 是什么

OCPP（Open Charge Point Protocol） 是由 OCA（Open Charge Alliance，开放充电联盟） 制定的一套开放通信协议，用于 充电桩（Charge Point） 与 中央管理系统（Central System / CSMS） 之间的通信。

一句话理解：OCPP 就是充电桩和云平台之间"说话"的标准语言。

OCPP 1.52012SOAP/XML增加预约充电、本地授权列表（LocalAuthList）、固件管理逐步淘汰

OCPP 1.6-J（J = JSON）是当前全球部署量最大的版本，绝大多数充电桩厂商都支持。
为什么 2.0.1 发布多年仍未普及？
1. 设备模型不兼容 — 2.0.1 从 Charge Point/Connector 改为 Charging Station/EVSE/Connector 三层模型，不能向下兼容 1.6，桩厂需要重新开发固件。
2. 认证测试滞后 — OCA 官方认证工具直到 2023 年才发布 2.0.1 测试套件，桩厂无法完成产品认证。
3. 投入回报不对称 — 1.6 已能满足绝大部分运营场景，升级到 2.0.1 需要桩+平台+测试全链条投入，但运营商从升级中获得的直接收益有限。
4. 存量设备庞大 — 全球已部署数百万个 OCPP 1.6 桩，运营商不会为了新协议替换硬件，平台必须长期兼容 1.6。
CHAPTER 02

为什么需要 OCPP

行业痛点：没有标准之前

在 OCPP 出现之前，充电行业面临严重的碎片化问题：

  • 每家桩企都有私有协议：ABB、Schneider、Delta 等厂商各自定义通信协议，互不兼容
  • 运营商被厂商绑定：选了某家的桩，就必须用他家的后台系统，想换桩？整个系统推倒重来
  • 互联互通困难：A 公司的 APP 找不到 B 公司的桩，用户体验极差
  • 开发成本高昂：每接入一种桩，就要重新开发一套对接适配层
  • 运维噩梦：不同协议的桩混合部署时，维护和升级极其复杂
类比理解
想象一下如果每个手机厂商都用自己的充电接口——这就是 OCPP 出现之前充电行业的状态。OCPP 就相当于充电行业的 USB-C 标准。
CHAPTER 03

OCPP 的核心优势

开放标准，厂商无关

  • 任何厂商都可以免费实现 OCPP
  • 运营商可以自由选择和混用不同品牌的充电桩
  • 避免厂商锁定（Vendor Lock-in）

降低对接成本

  • 云平台只需实现一套协议，即可接入市面上绝大多数充电桩
  • 新桩接入从"月级开发"降低到"天级配置"
  • 统一的消息格式，降低开发和调试难度

互联互通

  • 不同运营商的桩可以通过同一平台管理
  • 为漫游（Roaming）提供基础
  • 支持 OCPI、OICP 等漫游协议的底层前提

远程管理能力

  • 远程启停充电、远程固件升级
  • 远程修改配置、远程诊断
  • 实时监控桩的状态和充电数据

成熟且经过验证

  • 全球数百万充电桩在使用
  • 经过十多年的迭代和验证
  • 丰富的开源实现和测试工具

CHAPTER 05

国内外充电协议全景

充电桩行业存在多个通信协议标准，覆盖不同层级（桩↔平台、平台↔平台、桩↔车）和不同市场。理解这些协议的定位和关系，是做出正确技术选型的前提。

协议层级总览

层级功能国内协议国际协议
桩 ↔ 车充电握手、BMS通信GB/T 27930（CAN总线）ISO 15118（V2G）
桩 ↔ 平台设备管理、充电控制云快充、国网/南网私有OCPP 1.6 / 2.0.1 / 2.1
平台 ↔ 平台互联互通、漫游结算T/CEC 102OCPI
电网 ↔ 平台容量管理、需求响应—OSCP / OpenADR
关键认知：这些协议不是互相替代的关系，而是分层互补。OCPP 解决的是桩↔平台这一层的通信问题，和车辆通信（GB/T 27930 / ISO 15118）、平台互联（T/CEC 102 / OCPI）是不同层级的事。

一、桩 ↔ 平台协议（与 OCPP 同层级）

云快充协议

维度说明
传输方式TCP 长连接，二进制帧格式
最新版本v1.6.8+（持续迭代）
市场覆盖中国市场份额最大的桩↔平台协议，国内主流桩企均支持
开放程度需加入云快充生态获取协议文档，非完全开放标准

国网 / 南网私有协议

维度说明
使用方国家电网 e充电、南方电网 顺易充
特点基于国标扩展，增加平台管理功能，不对外公开
局限仅限自有体系使用，第三方无法接入

二、桩 ↔ 车协议

GB/T 27930

国标直流充电通信协议，基于 CAN 总线，用于充电桩与电动汽车 BMS 的握手和充电控制。所有国内直流桩必须实现。与 OCPP 是不同层级，互不替代。

ISO 15118

维度说明
关键特性Plug & Charge（插枪自动鉴权）、V2G 双向充电、动态价格调度
版本ISO 15118-2（单向）、ISO 15118-20（双向，2022 发布）
与OCPP关系互补。OCPP 2.0.1+ 集成了 ISO 15118 的证书管理和 Plug & Charge 流程

三、平台 ↔ 平台协议

中电联 T/CEC 102

中国电力企业联合会制定的平台互联互通协议，基于 HTTPS RESTful API。国家电网、特来电、星星充电等头部平台已对接，是国内跨平台找桩、充电、结算的事实标准。

OCPI（Open Charge Point Interface）

国际充电平台漫游互联协议，最新版本 OCPI 2.3（2024），功能与 T/CEC 102 类似但面向全球市场。支持跨运营商找桩、充电、结算。

四、电网 ↔ 平台协议

OSCP（Open Smart Charging Protocol）

电网发布可用容量，充电平台据此进行功率调度。与 OCPP SmartCharging 互补 — OSCP 提供容量约束，OCPP 执行功率分配。

OpenADR（Open Automated Demand Response）

电网需求响应协议（非充电专用），用于传递电价信号和削峰指令。可集成到充电平台实现电网友好充电。

CHAPTER 06

云快充：中国市场的事实标准

在桩↔平台通信这一层级，中国市场走出了一条与国际完全不同的路径。理解云快充为什么能成为国内事实标准，有助于理解 OCPP 在中国市场的定位。

时间线：为什么国内没用 OCPP

时间OCPP 状态中国市场状态
2010-2012OCPP 1.2/1.5 发布，SOAP/XML 格式，报文冗余、解析重，基于 HTTP 请求-响应模式，平台无法主动下发指令给内网桩国内充电桩行业刚起步，还未大规模建设
2014-2016OCPP 1.6 于 2015 年发布，引入 JSON/WebSocket，但国际生态尚未成熟国内充电桩爆发期，各桩企等不及，纷纷自研私有协议，互不兼容
2016-2018OCPP 1.6 在欧美逐步推广云快充公司抓住窗口期，推出 TCP 二进制协议 + SaaS 平台一体化方案，快速铺开
2018-至今OCPP 1.6 成为全球标准，2.0.1 发布云快充已占据市场，国内桩企无切换动力。出海桩企开始双协议支持
核心原因：OCPP 早期版本（1.2/1.5）技术不适用 + 中国市场爆发时间窗口不等人
OCPP 1.2/1.5 采用 SOAP/XML 格式，报文体积是 JSON 的 3-5 倍，对嵌入式设备内存和带宽都是负担。更致命的是基于 HTTP 请求-响应模式 — 桩在内网时平台无法主动下发指令（需要桩有公网 IP 或做端口映射）。这些技术缺陷让国内桩企在 2014-2016 年充电桩爆发期选择了自研方案。等到 OCPP 1.6 引入 JSON/WebSocket 解决了这些问题时，国内市场已经被云快充占据了。

云快充为什么能赢

产品层面

  • 不只是协议，是整套解决方案 — 协议 + SaaS平台 + 支付 + 运营工具，桩企接入即可运营
  • TCP 二进制，轻量高效 — 带宽效率高，嵌入式设备友好
  • 中文文档、中文技术支持 — 国内团队对接零障碍
  • 功能覆盖全面 — 充电/计费/运维/BMS 数据一应俱全

市场层面

  • 先发优势 — 在国内桩企最需要标准方案时出现，迅速建立网络效应
  • 无需 OCPP 的核心价值 — 国内市场不需要跨国互通，国标统一了充电接口，私有协议也够用
  • 监管无强制要求 — 中国的充电国标管桩↔车，桩↔平台通信没有强制标准
  • 生态壁垒 — 切换到其他协议意味着放弃现有平台和运营工具，成本高

OCPP vs 云快充：核心对比

对比维度OCPP 1.6-J云快充
标准组织OCA（国际开放联盟）云快充公司（商业主体）
协议开放性完全开放，可自由实现需加入生态获取文档
传输方式JSON over WebSocket二进制 over TCP
调试友好度JSON 可读，浏览器 F12 即可查看 WebSocket 帧二进制帧需专用工具抓包解析
海外市场欧盟 AFIR / 美国 NEVI 强制要求不被海外市场认可
国内市场头部桩企支持，但渗透率低于云快充国内主流，覆盖面广
第三方桩接入ABB、Schneider、Delta 等国际品牌原生支持仅国内桩企支持
扩展机制标准化 DataTransfer 消息，不破坏协议结构自定义帧类型扩展，各厂实现有差异

总结：云快充赢在国内生态和落地速度，OCPP 赢在开放性、标准化和国际通用性。如果只做国内市场，云快充更省事；如果要出海或接第三方桩，OCPP 是唯一选择。

趋势：双协议并行

  • 出海需求增长 — 国内桩企（华为、盛弘、英飞源、道通）出海时必须支持 OCPP，带动国内 OCPP 能力建设
  • 双协议成常态 — 越来越多桩企同时支持云快充 + OCPP，出厂时按目标市场选配
  • OCPP 2.1 成为 IEC 标准 — 有可能未来被纳入国内标准体系参考
CHAPTER 07

为什么选择 OCPP 1.6

选择 OCPP 1.6 的五个理由

理由一：海外市场硬性门槛
欧盟 AFIR（替代燃料基础设施法规）和美国 NEVI（国家电动汽车基础设施计划）均要求公共充电桩必须支持 OCPP。不支持 OCPP 意味着产品无法进入欧美市场。NeuronSystem 作为 AlwaysControl 的云平台，必须为硬件出海提供合规基础。
理由二：开放协议不被生态绑定
OCPP 是完全开放的标准，任何人都可以免费获取规范并实现。平台不依赖任何第三方服务商，不受商业生态约束。自主可控是云平台的核心诉求。
理由三：第三方桩接入能力
选择 OCPP 意味着平台天然兼容所有支持 OCPP 的充电桩。当企业客户要求接入他们已有的 ABB、Schneider、Wallbox 等品牌充电桩时，无需额外开发适配器。这是 SaaS 平台规模化的关键能力。
理由四：JSON/WebSocket 技术优势
JSON 格式人类可读，Chrome DevTools 即可调试 WebSocket 帧，开发效率远高于二进制协议。WebSocket 天然穿越 NAT 和防火墙，适合家庭场景下 N3 Lite 从内网连接云平台。
理由五：清晰的技术演进路径
OCPP 1.6 → 2.0.1 → 2.1 是一条明确的升级路线。2.1 已成为 IEC 63584-210:2025 国际标准，支持 V2G、ISO 15118-20、DER 等下一代能源管理特性。选择 OCPP 就是选择与行业标准同步演进。
为什么现阶段选 1.6 而不是直接上 2.0.1？
务实选择。OCPP 1.6 已经能完整覆盖 NeuronSystem 当前阶段的全部需求：设备注册、充电控制、遥测上报、智能充电（DLM）、固件升级。2.0.1 的核心新增（ISO 15118 Plug & Charge、三层设备模型、安全证书管理）在 NeuronSystem Phase 1-3 阶段均不需要。

更重要的是，95% 以上的充电桩只支持 1.6，如果平台只实现 2.0.1，将无法接入绝大多数第三方桩。

NeuronSystem 的策略是：以 OCPP 1.6 交付、以 2.0.1 架构设计。neuron-protocol 模块已预留版本协商能力，当桩端固件升级到 2.0.1 时，平台可平滑支持。

CHAPTER 08

我们的云平台为什么选择 OCPP

8.1 海外市场的硬性门槛

OCPP 是出海的"入场券"

  • 欧洲、北美、澳洲等主流市场，OCPP 兼容性是充电运营商的基本采购要求
  • 欧盟 AFIR（Alternative Fuels Infrastructure Regulation）法规明确要求充电基础设施支持开放协议
  • 美国 NEVI（National Electric Vehicle Infrastructure）联邦补贴项目，强制要求充电桩支持 OCPP 1.6 或以上
  • 不支持 OCPP = 无法参与海外招投标，直接被排除在市场之外
关键结论：不支持 OCPP = 放弃海外市场。这不是技术选型的偏好问题，而是市场准入的硬性要求。

海外客户的真实诉求

  • 海外运营商管理的桩来自多个品牌，他们需要一个平台能统一管理所有 OCPP 桩
  • 运营商不会为了接入某一家的桩去部署那家的私有后台——这在海外是不可接受的
  • 支持 OCPP 意味着 我们云平台 可以作为第三方运营平台，接入任意品牌的标准桩

8.2 巨大的市场空间与发展前景

全球充电桩爆发式增长

  • 全球公共充电桩保有量已超过 400 万台（2024）
  • IEA 预测到 2030 年全球需要 1500 万+ 公共充电桩
  • 欧洲 2035 年禁售燃油车，充电基础设施需求将持续井喷
  • 东南亚、中东、南美等新兴市场正在起步，是蓝海

OCPP 的市场渗透率

  • 全球新部署的充电桩中，超过 80% 支持 OCPP
  • OCPP 1.6 是当前绝对主流，市场占有率最高
  • OCPP 2.0.1 正在逐步推广，但 1.6 仍将长期共存（至少 5-8 年）
  • 支持 OCPP 1.6 + 2.0.1 双版本，可以覆盖几乎所有海外充电桩
充电运营是长期生意：充电桩不是一次性买卖，运营商需要持续的平台服务（SaaS 模式）。每接入一台桩 = 一个长期付费客户。平台的价值随接入桩数增长而增长（网络效应）。

8.3 适配 OCPP 后的云平台战略价值

从硬件商到平台商的跃迁

商业模式升级
只卖硬件（N3 / N3Lite）：利润有限，市场天花板低。
硬件 + 平台：通过 OCPP 接入第三方桩，我们云平台 变成充电运营平台。
平台可以输出给海外运营商、物业、车队管理公司等，商业模式从"卖设备"升级为"卖服务"。

互联互通的基础

  • 支持 OCPP 后，可以进一步对接 OCPI（Open Charge Point Interface） 漫游协议
  • 实现跨运营商的充电漫游——用户用一个 APP 在多个运营商的桩上充电
  • 这是海外充电网络的核心需求，也是平台估值的关键指标

8.4 技术层面的务实选择

为什么先做 OCPP 1.6 而不是直接上 2.0.1

  • 1.6 是存量最大的版本：全球数百万台桩在用，接入即有客户
  • 1.6 实现复杂度适中：消息集固定（28 个），协议文档成熟，开源参考多
  • 2.0.1 向下兼容性好：1.6 的核心概念在 2.0.1 中都保留了，先做 1.6 的架构设计可以平滑升级
投入产出比：实现 OCPP 1.6 的核心流程（注册 -> 鉴权 -> 充电 -> 计费）。完成后即可接入市面上绝大多数充电桩，一次开发，长期受益。相比为每个厂商的私有协议做适配，OCPP 的 ROI 高出一个数量级。

以下是一个充电桩从首次上线到完成一笔充电交易的完整生命周期。

阶段一：建立连接

充电桩                                              云平台
  |                                                   |
  |  ---- WebSocket 握手 (wss://host/ocpp/CP001) -->  |
  |                                                   |
  |  <---- WebSocket 101 Switching Protocols --------  |
  |                                                   |
  |            [ WebSocket 长连接建立 ]                  |
关键点：桩上电后，根据配置的 URL 主动发起 WebSocket 连接。URL 中包含桩的身份标识（ChargePointIdentity）。云平台可以在握手阶段通过 HTTP Header 做初步认证，支持 Basic Auth 或 TLS 客户端证书认证。

阶段二：启动注册（BootNotification）

充电桩                                              云平台
  |                                                   |
  |  ---- BootNotification.req ------------------->   |
  |       {                                           |
  |         chargePointVendor: "AlwaysControl",       |
  |         chargePointModel: "ATP-III-AC-7kW",       |
  |         chargePointSerialNumber: "ACT20240001",   |
  |         firmwareVersion: "1.2.0",                 |
  |         meterType: "ACM100",                      |
  |         meterSerialNumber: "MTR20240001"          |
  |       }                                           |
  |                                                   |
  |  <---- BootNotification.conf -------------------  |
  |       {                                           |
  |         status: "Accepted",                       |
  |         currentTime: "2025-03-05T10:00:00Z",      |
  |         interval: 300                              |
  |       }                                           |

BootNotification 的三种响应状态：

状态含义桩的行为
Accepted注册成功正常工作，按 interval 发心跳
Pending等待审核等待 interval 后重新发送 BootNotification，不接受充电请求
Rejected拒绝注册等待 interval 后重新发送 BootNotification
要点：BootNotification 是桩连接后必须发送的第一条消息。云平台通过 interval 告知桩心跳间隔（秒），通过 currentTime 同步桩的时钟。如果状态是 Rejected，桩只能发 BootNotification，不能做其他操作。

阶段三：状态上报（StatusNotification）

connectorId 说明：0 代表充电桩整体，1, 2, ... 代表各个充电枪/连接器。

CHAPTER 13

断网补偿机制

充电桩部署在户外环境，网络中断是常态而非异常。OCPP 1.6 在协议层面定义了完整的离线行为规范，确保断网不影响用户充电，恢复后数据零丢失。

13.1 断网场景分类

场景触发条件影响范围桩端行为
A. 充电前断网用户刷卡时 WebSocket 已断开鉴权无法在线完成使用 LocalAuthList 本地鉴权
B. 充电中断网充电进行中网络中断MeterValues 无法上报继续充电，本地缓存电表数据
C. 结束时断网充电结束时仍未恢复网络StopTransaction 无法发送本地存储交易记录，恢复后补发
D. 长时间断网网络持续中断数小时多笔交易积压持续缓存，按队列顺序补发

13.2 桩端离线能力

本地鉴权（LocalAuthList）

云平台通过 SendLocalList 消息预先将授权用户名单下发到桩的本地存储：

云平台                                              充电桩

  |         updateType: "Full",                       |

  |           { idTag: "USER_001",                    |
  |             idTagInfo: { status: "Accepted",      |
  |               expiryDate: "2025-12-31T23:59:59Z"  |
  |           } },                                    |
  |           { idTag: "USER_002",                    |
  |             idTagInfo: { status: "Accepted" } },  |
  |           ...                                     |
  |       ] }                                         |
  |                                                   |

  |       { status: "Accepted" }                      |
离线鉴权流程：用户刷卡 → 桩检测到 WebSocket 断开 → 查询本地 LocalAuthList → 匹配到 idTag 且未过期 → 允许充电。名单支持全量更新（Full）和增量更新（Differential）。云平台应定期同步，保持名单新鲜度。

AuthorizationCache（鉴权缓存）

除了主动下发的 LocalAuthList，桩还可以启用 AuthorizationCacheEnabled 配置项，自动缓存在线鉴权成功的 idTag 结果。两者的区别：

特性LocalAuthListAuthorizationCache
数据来源云平台主动下发桩自动缓存在线鉴权结果
更新方式SendLocalList 指令每次 Authorize 响应自动更新
优先级高（优先查询）低（LocalAuthList 未命中才查）
清除方式SendLocalList 空列表ClearCache 指令

离线交易缓存

桩在离线状态下产生的交易数据必须持久化存储，OCPP 1.6 通过以下配置项控制：

配置项含义建议值
LocalAuthorizeOffline离线时是否允许本地鉴权true
LocalPreAuthorize在线时是否先查本地再查云端true（减少网络延迟）
AllowOfflineTxForUnknownId离线时是否允许未知 idTag 充电false（安全考虑）
MaxEnergyOnInvalidId无效 idTag 最大允许充电量 (Wh)0（不允许）
StopTransactionOnInvalidId在线鉴权失败是否停止充电true

13.3 云平台侧补偿设计

设计原则：桩端负责"不丢数据"，云平台负责"数据一致性"。桩断网恢复后会补发所有积压的消息，云平台必须能正确处理这些延迟到达的、可能乱序的历史数据。

云平台需要处理的补偿场景

  1. 延迟到达的 StartTransaction：桩恢复后补发，云平台应根据 timestamp 字段回溯创建交易，而非使用收到消息的当前时间
  2. 延迟到达的 StopTransaction：交易可能在云平台已标记为"异常未结束"，收到补发后需要正确结算
  3. 批量到达的 MeterValues：恢复后短时间内涌入大量历史电表数据，需要按 timestamp 排序后入库
  4. 交易 ID 为负数：离线期间桩无法从云平台获取 transactionId，会使用桩自行生成的本地 ID（通常为负数），云平台收到后应分配正式 ID 并建立映射
关键实现点：云平台在处理 StartTransaction 时，如果发现 timestamp 远早于当前时间（如超过 5 分钟），应标记为"离线补报交易"，走独立的校验和入账逻辑，避免影响实时交易的处理流程。
CHAPTER 14

并发与重试逻辑

OCPP 1.6 的消息模型有严格的并发控制规则，理解这些规则是正确实现服务端的基础。

14.1 消息并发模型

核心规则：OCPP 1.6 规定，在同一方向上（CP→CS 或 CS→CP），同一时刻只能有一个 CALL 消息等待响应。发送方必须等到收到 CALLRESULT 或 CALLERROR 后，才能发送下一个 CALL。

双向并发是允许的

  CP → CS: BootNotification.req  (等待响应中...)
  CS → CP: GetConfiguration.req  (允许! 不同方向互不阻塞)
  CP → CS: Heartbeat.req         (不允许! 必须等 BootNotification 响应)

云平台并发处理设计

虽然协议限制桩端是串行发送的，但云平台同时管理成百上千台桩，需要高并发处理：

维度设计要点
连接管理每个 WebSocket 连接独立的消息队列，桩之间完全并行
下行队列云平台对每台桩维护一个 FIFO 队列，确保同一时刻只有一条下行 CALL 在等待
响应匹配通过 CALL 中的 UniqueId 匹配响应，使用 ConcurrentHashMap<String, CompletableFuture> 实现
线程模型Netty EventLoop 处理 I/O，业务逻辑投递到独立线程池，避免阻塞 I/O 线程

14.2 超时处理

OCPP 1.6 未规定统一的超时时间，但实际部署中必须设定合理的超时：

超时场景建议超时超时后行为
桩发送 CALL，等待云平台响应30 秒桩可以重发或丢弃该消息（交易相关消息必须重发）
云平台发送 CALL，等待桩响应30 秒视为桩未执行，根据业务决定是否重发
WebSocket 空闲（无任何消息）Heartbeat interval × 2 + 30s云平台判定桩离线，关闭连接
WebSocket Ping/Pong10 秒Pong 未收到则关闭连接
实现建议：使用 Netty 的 IdleStateHandler 检测读空闲。当读空闲超过 heartbeatInterval × 2 + 30s 时触发 userEventTriggered，主动关闭连接并将桩标记为离线。

14.3 重试策略

桩端重试（交易相关消息）

首次发送 → 等待 30s → 超时
    ↓
第 1 次重试 → 等待 TransactionMessageRetryInterval → 超时
    ↓
第 2 次重试 → 等待 TransactionMessageRetryInterval → 超时
    ↓
    ... 共 TransactionMessageAttempts 次
    ↓
全部失败 → 消息仍保留在持久化队列，等待 WebSocket 恢复后继续

云平台重试（下行指令）

云平台下发的指令（如 RemoteStartTransaction）重试需要更加谨慎：

指令类型可否重试说明
RemoteStartTransaction可重试桩会检查状态，已在充电则返回 Rejected，幂等安全
RemoteStopTransaction可重试桩找不到 txId 则返回 Rejected，幂等安全
SetChargingProfile可重试覆盖式写入，幂等
Reset谨慎重试可能导致桩多次重启，建议最多 1 次重试
UpdateFirmware谨慎重试重复下载消耗带宽，桩会检查版本，但仍应限制重试
UnlockConnector可重试机械操作，失败可能是硬件问题，重试 1 次

14.4 幂等性设计

云平台接收消息必须保证幂等：桩可能因为超时重发消息，云平台收到重复的 StartTransaction 或 StopTransaction 时，不能重复创建交易或重复结算。

去重策略

  • StartTransaction 去重：以 chargePointId + connectorId + idTag + timestamp 组合为唯一键，相同组合视为重复消息，返回相同的 transactionId
  • StopTransaction 去重：以 transactionId 为键，已结束的交易收到重复 Stop，返回成功但不重复扣款
  • MeterValues 去重：以 transactionId + timestamp + connectorId 为键，相同时间点的数据跳过
CHAPTER 15

安全性设计

充电桩涉及电力控制和资金交易，安全性直接关系到人身安全和财产安全。

15.1 安全配置等级（Security Profiles）

OCPP 1.6 安全白皮书定义了三个安全等级，逐级递增：

等级传输安全身份认证适用场景
Profile 1TLS (wss://)Basic Auth（HTTP Header 中的用户名密码）入门级部署，中小规模
Profile 2TLS (wss://)TLS 服务端证书 + Basic Auth标准商业部署
Profile 3Mutual TLS (mTLS)双向证书认证（桩和平台各持证书）高安全要求场景
NeuronCloud 建议：初期采用 Profile 2（TLS + Basic Auth），兼顾安全性与部署便利性。Profile 3 的双向证书管理在桩数量大时运维成本较高，待平台规模增长后再升级。

15.2 传输层安全（TLS）

WebSocket 握手中的认证

GET /ocpp/ACT-N3-20240001 HTTP/1.1
Host: charge.alwayscontrol.net:9001
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Protocol: ocpp1.6
Authorization: Basic base64(chargePointId:password)

TLS 配置建议

配置项建议值说明
TLS 版本≥ TLS 1.2禁用 TLS 1.0/1.1（已知漏洞）
加密套件TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 等使用 AEAD 加密，禁用 CBC 模式
证书RSA 2048-bit 或 ECC P-256正式环境使用 CA 签发证书
HSTS启用防止降级攻击

15.3 应用层安全

WebSocket 握手阶段

  1. ChargePointIdentity 校验：URL 中的标识必须是已注册的合法设备，未注册则拒绝握手（返回 HTTP 404）
  2. Sec-WebSocket-Protocol 校验：必须包含 ocpp1.6，否则拒绝（返回 HTTP 400）
  3. Basic Auth 密码校验：密码与平台记录不匹配则拒绝（返回 HTTP 401）
  4. IP 白名单/黑名单：可选，对已知恶意 IP 直接拒绝
  5. 连接速率限制：同一 IP 在短时间内的连接次数限制，防止 DDoS

消息层校验

校验项规则违规处理
JSON 格式必须是合法 JSON 数组，长度 3 或 4 或 5返回 CALLERROR: FormationViolation
MessageTypeId必须是 2, 3, 或 4返回 CALLERROR: ProtocolError
Action 名称必须是已知的 OCPP 消息名返回 CALLERROR: NotImplemented
Payload Schema字段类型和约束符合 JSON Schema返回 CALLERROR: TypeConstraintViolation
消息大小单条消息 ≤ 64KB（可配置）断开连接
UniqueId 长度≤ 36 字符返回 CALLERROR: PropertyConstraintViolation

15.4 安全事件监控

应当触发安全告警的事件：
  • 同一桩短时间内多次 BootNotification（可能被重放攻击）
  • 未注册的 ChargePointIdentity 尝试连接（扫描行为）
  • Basic Auth 密码连续失败 ≥ 3 次（暴力破解）
  • 消息频率异常（正常桩不会每秒发多条消息）
  • WebSocket 握手中携带异常 Header（注入攻击）
  • 接收到不符合 OCPP 规范的消息格式（协议模糊测试/攻击）
为什么 OCPP 1.6 的安全设计很重要？
充电桩是连接电网的大功率设备。如果攻击者能通过伪造消息控制充电桩，轻则造成用户财产损失（免费充电、多扣费），重则可能导致电力安全事故（超负荷运行、短路、火灾）。云平台作为所有桩的中央控制节点，安全等级应等同于关键基础设施。
