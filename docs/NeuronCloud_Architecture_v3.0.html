概要总览

我们要做什么

为智能网关 N3 Lite 和充电设备构建统一的云端管理平台。 第一步打通 N3 Lite 数据通道: 实现设备上线、数据沉淀、上传下挂充电桩信息、云端配置 DLM 电流策略。 所有设备通过 OCPP 1.6 统一接入，N3 Lite 在云平台眼中就是一台"多枪充电桩"。 数据跑通后，逐步扩展家庭充电控制能力，进一步实现企业计费运营能力。

核心设计理念
  • 数据通道先行: 先让设备连上来、数据看得到，再做业务功能
  • 单协议统一: 所有设备通过 OCPP 接入，业务层只写一套代码
  • 数据零丢失: 关键数据 OCPP 持久化 + Kafka + 幂等消费 多层保障
  • 离线安全: N3 Lite 本地 DLM 和充电控制不依赖云端
  • 渐进式交付: 数据通道 → 家庭充电 → 企业运营，每一步都可独立交付

业务背景

现状: 每台设备都是一座孤岛

AlwaysControl 是一家充电设备制造商，核心产品包括充电桩 (ATP III) 和智能网关 (N3 Lite)。这些设备主要安装在家庭用户家中，但目前有一个共同的问题: 卖出去之后，我们和设备之间就"失联"了。

现状痛点具体影响业务代价
设备离线用户出门后无法知道充电进度，不能远程启停用户体验差，产品竞争力不足
无法远程诊断设备报故障，售后团队无从判断原因每一次售后都是上门服务，成本高、响应慢
数据无沉淀不知道设备运行多少小时、充了多少电产品迭代缺乏数据支撑，用户没有充电记录
固件无法远程更新发现 Bug 或加新功能，需要上门刷机新功能推送周期长，Bug 修复成本高
负载无法管理家庭进线容量有限，多桩同充可能跳闸用户投诉，硬件口碑受损
核心问题: 没有云平台，AlwaysControl 只是一个"卖硬件的公司"。在充电桩行业竞争日益激烈的今天，没有配套 APP 和云管理的充电产品已经不具备竞争力。

目标: 三步走

第一步: 打通数据通道 — 设备上线，数据可见 当前

最优先的事情: 让 N3 Lite 连上云平台，让我们看到数据。

能力说明价值
设备注册与在线N3 Lite 通过 OCPP 连接云平台，自动注册，实时心跳知道每台设备在不在线

实时数据上报CT 电流/电压/功率、各桩状态、温度 → 数据沉淀到数据库设备运行状态有据可查
DLM 电流配置云端下发 DLM 策略 (总容量/模式/安全系数) → N3 Lite 本地执行远程调整负载管理参数
异常告警设备故障/过温/通信断开 → 推送通知问题第一时间知道
远程 OTA新功能和 Bug 修复远程推送产品持续进化，降低售后成本
第一步的核心交付: N3 Lite 连上云 → 后台看到设备树和实时数据 → 能远程配置 DLM 参数 → 能远程 OTA。
不涉及: 用户 App、充电控制、支付计费。这一步面向的是 AlwaysControl 内部团队和售后人员。

第二步: 家庭充电管家 — 用户能控制充电 下一步

能力说明价值
远程启停充电App 一键启停，查看实时功率和电量出门在外也能管理充电
充电记录每次充了多久、多少度电，历史可追溯用户清楚每一笔用电
用电统计CT 数据可视化，充电桩功率占比帮用户了解家庭用电情况

家庭用户的充电流程: 打开 App → 点"开始充电" → 看实时状态 → 点"停止" → 查看记录。不需要扫码，不需要付费 — 这是自己家的桩、自己家的电。

第三步: 商业运营平台 — 对企业开放 远期规划

当家庭充电能力成熟后，可进一步扩展面向企业的商业运营能力：

受益方有了什么能力商业价值
企业运营商扫码充电、按量计费、多站点管理、收入报表SaaS 运营收入，平台抽成
AlwaysControl硬件 + SaaS 双收入模式从一锤子买卖到持续收入
家庭用户升级开放自家桩给邻居用 → 扫码收费家庭用户秒变微型运营商

两种租户，一套系统

家庭租户 (HOME): 买了 N3 Lite + 充电桩的业主。绑定设备 → 直接控制 → 看记录。不走支付流程。

企业租户 (BUSINESS): 有桩/有场地的运营商，接入平台做充电运营。扫码 → 鉴权 → 充电 → 结算。

技术实现: tenant.type = HOME / BUSINESS，按类型控制功能开放。充电控制逻辑通用 (OCPP)，计费模块按类型开关。
升级路径: 家庭用户想开放共享充电？改 type + 补充商户号 = 秒变运营商。

商业模式

阶段一 (家庭): 卖硬件 (N3 Lite + ATP III) + 可选年费增值服务 (高级能耗分析、定时充电策略)。

阶段二 (企业): 硬件 + SaaS 服务费抽成。用户扫码付款，电费归运营商，平台抽取服务费。具体支付渠道和分账方案根据目标市场确定。

硬件是入口，云平台是护城河。 每多卖一台设备，平台价值增加一分。
设计理念

五条核心设计理念

理念一: 数据通道先行

平台的第一要务是让设备连上来、让数据看得到。N3 Lite 通过 OCPP WebSocket 连接云平台，上报设备拓扑、CT 电流、桩状态、DLM 分配等全量数据。数据沉淀到数据库后，才有后续一切业务功能的基础。

为什么不直接从充电控制或计费开始做？
充电控制和计费都依赖稳定的数据通道。如果连 N3 Lite 的连接、注册、数据上报都没跑通，做任何上层业务都是空中楼阁。先把通道打通、数据沉淀做扎实，上层功能可以快速叠加。

理念二: 单协议统一 — OCPP 覆盖所有设备

不管充电桩是自己直连云平台，还是通过 N3 Lite 代理接入，云平台看到的都是标准的 OCPP Charge Point。设备管理、数据采集、充电控制等所有业务模块只需要写一套代码。

为什么不用 MQTT 做 IoT 数据上报？
N3 Lite 是 ESP32 设备，维护两个协议栈 (WebSocket + MQTT Client) 消耗更多内存和复杂度。OCPP 已经能覆盖所有需求:
充电事务: Start/StopTransaction、MeterValues — OCPP 原生
CT 电流监测: connector 0 的 MeterValues (Current.Import L1/L2/L3) — OCPP 合规用法
DLM 状态: DataTransfer (vendorId: AlwaysControl) — OCPP 标准扩展方式
OTA: UpdateFirmware — OCPP 原生

施耐德 EVlink Pro AC、ABB Terra 集群控制器也是同样的做法 — 一条 OCPP 通道搞定所有事情。一个协议意味着更少的代码、更少的 bug、更少的运维。

理念三: 关键数据零丢失

充电记录、计费数据 — 一旦丢失直接影响用户信任和商业结算。OCPP 协议在设计层面就保证了交易数据的持久化和补传能力。

具体怎么做到零丢失？
OCPP 1.6 对交易相关消息 (StartTransaction / StopTransaction / MeterValues with txId) 有明确的持久化和重传要求。设备端必须将这些消息写入本地存储，断网恢复后按序补发。配合云平台的 Kafka 持久化 + 消费端幂等处理，形成端到端的零丢失链路。OCPP 保证的是业务层送达，比传输层可靠性更强。

理念四: 离线安全，来网即同步

N3 Lite 在断网时必须能独立工作: 本地 DLM 负载均衡继续运行，充电桩正常充电，交易数据本地缓存。恢复连接后自动将缓存数据按序补发给云平台。

断网了用户还能充电吗？
可以。OCPP 1.6 原生支持 LocalAuthList — 云平台预先将授权用户名单下发到设备本地存储。断网时使用本地名单鉴权，充电正常进行，数据在本地缓存。联网后自动同步。

理念五: 渐进式交付，模块化单体

平台分三步交付: 数据通道 → 家庭充电 → 企业运营。每一步都是可独立交付的完整产品。技术上，所有模块部署在一个 JVM 进程中，每个模块是独立的 Maven module，边界清晰，未来可按需拆分为微服务。

设备与接入

路径 A: 充电桩直连

充电桩内置 OCPP 客户端，通过 WebSocket 直连云平台。适用于有独立网络的商业充电站。

  • 连接 wss://charge.alwayscontrol.net:9001/ocpp/{chargePointId}
  • 完整支持 OCPP 1.6 全部消息
  • 一台双枪桩 = connectorId 0 (本体) + connectorId 1, 2 (两个枪)

路径 B: N3 Lite 虚拟桩

N3 Lite 是一台基于 ESP32 的智能网关，对云平台来说它就是一台"多枪充电桩"。每台下挂充电桩的每个枪都映射为一个 connectorId。

Connector 映射 — 以 2 台双枪桩为例

connectorId映射到数据内容
0N3 Lite 本体CT 电流 (L1/L2/L3)、家庭总功率、温度 — 通过 MeterValues 上报
1充电桩 A - 枪 1充电事务 + 电表数据
2充电桩 A - 枪 2充电事务 + 电表数据
3充电桩 B - 枪 1充电事务 + 电表数据
4充电桩 B - 枪 2充电事务 + 电表数据

N3 Lite 注册后通过 DataTransfer 上报拓扑映射:

  "vendorId": "AlwaysControl",
  "messageId": "ConnectorTopology",

    {\"id\":\"ATP-A001\",\"model\":\"ATP-III-7kW-2C\",\"connectors\":[1,2]},
    {\"id\":\"ATP-B002\",\"model\":\"ATP-III-7kW-2C\",\"connectors\":[3,4]}
  ]}"
}]
云平台怎么区分"直连桩"和"N3 Lite"？
在 OCPP 层面不需要区分 — 充电管理、计费、远程控制的逻辑完全一样。但在设备管理层面，可以通过 BootNotification 中的 chargePointModel: "N3-Lite" 识别，在设备表中标记 deviceType = GATEWAY，并存储 connector 拓扑映射。
如果下挂桩动态增减怎么办？
N3 Lite 通过 PLC 发现新桩或检测到桩离线时，重新发送 DataTransfer (ConnectorTopology) 更新拓扑，并为新增的 connector 发送 StatusNotification。云平台更新设备拓扑表即可。
架构总览

六层架构

展示层
用户 App (iOS + Android, 家庭: 远程控制 / 企业: 扫码充电) | 管理后台 (Vue3 + Ant Design Vue) | 数据大屏 (可选)
接口层
REST API (Spring Boot 3.x) — CRUD 与业务操作 | WebSocket — 实时数据推送至 APP
业务层
设备管理 | 能耗分析 | 告警中心 | OTA 升级 | 充电管理 (Phase 2) | 计费结算 (Phase 3)
消息层
Kafka — 按类型分 Topic: billing / telemetry / event / command (缓冲 / 解耦 / 可重放)
设备接入层
OCPP Gateway (Netty WebSocket, 端口 9001) — 统一接入直连桩和 N3 Lite，解析消息 → 统一格式 → Kafka
基础设施层
PostgreSQL (业务 + TimescaleDB 时序) | Redis (缓存 / 在线状态) | MinIO (对象存储) | 日志与监控 (Logback / ELK)
为什么设备接入层只有一个组件？
所有设备 (直连桩 + N3 Lite) 都通过 OCPP 协议接入同一个 Netty WebSocket 服务，不需要额外的消息中间件 (如 MQTT Broker)。接入层的职责: 把 OCPP 消息翻译成统一的内部格式，投递到 Kafka。业务层只消费 Kafka 消息，完全不关心消息来自哪种设备。
消息层 (Kafka) — 为什么不直接写数据库？
设备消息高频、突发、不可控。Kafka 在中间起三个作用:
1. 削峰 — 突发流量先进队列，消费端按自己的节奏处理
2. 解耦 — 一条消息可被多个消费者处理 (存时序库的、触发告警的、推送 APP 的)
3. 可重放 — 消费端出错可重新消费，不丢数据
模块设计

平台模块

Phase 1: 数据通道 — "设备上线, 数据可见" 当前

neuron-protocol
OCPP 协议网关
Netty WebSocket 服务 (端口 9001)。接收所有 OCPP 连接 (直连桩 + N3 Lite)。解析标准消息 + DataTransfer 扩展消息 → 统一格式 → Kafka。下行指令 (DLM 配置、OTA) → 转 OCPP → 发送给设备。
neuron-device
设备管理
设备注册 (BootNotification 自动注册)、在线状态 (Heartbeat + Redis TTL)、设备拓扑 (N3 Lite connector → 物理桩/枪映射)、配置管理 (DLM 电流策略下发)、OTA 编排。
neuron-energy
数据采集与能耗
接收并存储 CT 电流/电压/功率数据 (connector 0 MeterValues)、各 connector 状态、DLM 分配记录。数据可视化: 实时数据看板、历史趋势图。
neuron-alert
告警中心
告警规则引擎、分级告警记录、推送通知。数据来源: StatusNotification (Faulted) + DataTransfer (DLM 过载告警) → Kafka[event]。
neuron-system
系统基座
后台管理 (用户/角色/权限)、租户管理 (tenant.type = HOME / BUSINESS)、数据字典。Shiro 2.0.5 + JWT 认证授权。
Phase 1 的核心交付是什么？
数据通道闭环: N3 Lite 能连上来 (protocol) → 知道有哪些设备、下挂了哪些桩 (device) → 实时看到 CT/功率/状态数据 (energy) → 设备故障有告警 (alert) → 有人能登录后台管理 (system)。

简单控制能力: 云端可以通过 DataTransfer 下发 DLM 电流配置 (总容量/模式/安全系数)，N3 Lite 在本地执行负载分配。

不包含: 用户 App、远程启停充电、支付计费。这一步面向 AlwaysControl 内部团队 — 先确保数据通道稳定可靠。

Phase 2: 家庭充电 — "用户能远程控制充电" 下一步

neuron-charge
充电管理
充电会话状态机 (启动/充电中/停止/完成)、充电记录、远程控制 (启/停/设电流)。家庭用户直接 RemoteStart — 不涉及计费。消费 Kafka 处理 OCPP 交易消息。
neuron-user
用户管理
App 登录 (Google / Apple Sign-In)、设备绑定 (用户→N3 Lite→connector)、idTag 分配。家庭用户绑定自己的设备后直接控制，不需要扫码。
Phase 2 在 Phase 1 基础上加了什么？
Phase 1 已经有了稳定的数据通道和设备管理。Phase 2 加两件事:
1. 充电控制 (charge): 让云平台具备远程启停充电桩的能力，记录每次充电的电量和时长。
2. 用户入口 (user): 开放给家庭用户的 App，绑定设备后一键控制。

不涉及支付体系。 家庭用户用的是自己家的桩、自己家的电，不需要付费。

Phase 3: 企业运营 — "扫码充电, 按量计费" 后续

neuron-billing
计费结算
电价策略 (分时段)、计费引擎 (基于 OCPP MeterValues)、预授权/扣款/账单生成、第三方支付对接 (具体渠道待定)。只对 BUSINESS 租户启用。
neuron-operator
运营管理
运营商入驻与审核、站点管理、收入报表、分账配置、商户号绑定。
Phase 3 和 Phase 2 的关系？
Phase 2 的充电控制 (charge) 模块对所有租户通用。Phase 3 在上面加一层计费 (billing) 和一层运营商管理 (operator)。billing 消费 charge 通过 Kafka 发出的充电事件，按电价策略计费结算。不改 Phase 1 和 Phase 2 的任何代码。

N3 Lite 虚拟桩、数据通道与充电流程

平台最核心的技术设计。

N3 Lite 架构

N3 Lite 虚拟桩: OCPP 全量覆盖

N3 Lite 通过一条 OCPP WebSocket 连接完成所有数据交互。参考施耐德 EVlink Pro AC 集群控制器的做法 — 本地控制器注册为一台 OCPP Charge Point，下挂的实际充电桩映射为 connector。

OCPP 消息通道映射

N3 Lite (OCPP Charge Point "N3L-001")
│
├── 标准 OCPP 消息 (充电业务)
│   ├── BootNotification          注册, 上报 model="N3-Lite", connector 数
│   ├── Heartbeat                 保活
│   ├── StatusNotification        各 connector 状态 (Available/Charging/Faulted)
│   ├── Authorize                 用户鉴权 (在线/LocalAuthList)
│   ├── StartTransaction          充电开始, 电表起始读数
│   ├── MeterValues (conn 1~N)    充电中电表 (Energy/Power/Current/Voltage)
│   ├── StopTransaction           充电结束, 电表终止读数
│   └── ◄── RemoteStart/Stop      云平台远程控制
│
├── connector 0 MeterValues (CT/家庭数据, OCPP 合规用法)
│   ├── Current.Import L1/L2/L3   CT 监测: 家庭进线三相电流
│   ├── Power.Active.Import       家庭总有功功率
│   ├── Voltage L1/L2/L3          进线电压
│   └── Temperature               N3 Lite 本体温度
│
├── DataTransfer 上行 (vendor 扩展)
│   ├── ConnectorTopology          connector → 物理桩/枪映射关系
│   ├── DLMStatus                  DLM 分配: 各桩电流限值 / 可用容量 / 模式
│   └── DLMAlert                   DLM 过载保护触发事件
│
├── DataTransfer 下行 (云 → N3 Lite)
│   ├── DLMConfig                  DLM 策略: 总容量 / 模式 / 安全系数
│   └── DeviceCommand              N3 Lite 本体控制指令
│
└── 标准 OCPP 下行 (云 → N3 Lite)
    ├── SetChargingProfile         功率限制 / DLM 电流限值
    ├── ChangeConfiguration        参数配置
    └── UpdateFirmware             OTA 固件升级

CT 数据上报: connector 0 MeterValues

OCPP 1.6 规定 connectorId = 0 代表 Charge Point 本体。N3 Lite 利用这个标准机制上报家庭 CT 数据:

[2, "mv-ct-001", "MeterValues", {
  "connectorId": 0,
  "meterValue": [{
    "timestamp": "2026-03-11T10:00:00Z",
    "sampledValue": [
      {"value": "28.5", "measurand": "Current.Import", "phase": "L1", "unit": "A"},
      {"value": "31.2", "measurand": "Current.Import", "phase": "L2", "unit": "A"},
      {"value": "27.8", "measurand": "Current.Import", "phase": "L3", "unit": "A"},
      {"value": "6120", "measurand": "Power.Active.Import", "unit": "W"},
      {"value": "230.5", "measurand": "Voltage", "phase": "L1", "unit": "V"}

这不是 hack，是合规用法。 OCPP 1.6 规范明确支持 connector 0 的 MeterValues，用于上报 Charge Point 级别的计量数据。Current.Import / Power.Active.Import / Voltage / Temperature 都是标准 measurand。

性能分析: 一条通道够不够？

OCPP 同一方向同时只能有一个 CALL 等待响应。以最忙场景 (4 枪同时充电) 测算:

消息类型频率条/分钟
Heartbeat每 300s0.2
CT MeterValues (conn 0)每 60s1
充电 MeterValues (conn 1-4)每 60s × 44
DLM DataTransfer每 60s1
合计~7 条/分钟

按最差 500ms/条 RTT 计算: 7 × 0.5 = 3.5 秒/分钟通信时间。空闲率 94%，毫无压力。

ESP32 资源够用吗？
单通道 (WebSocket only) 占用约 15-20KB RAM。如果做双通道 (WebSocket + MQTT)，需要 30-40KB。ESP32 可用 SRAM 约 200KB，单通道省下来的 15-20KB 可以用于离线交易缓存，对嵌入式来说是实打实的优势。

代码怎么区分两种流程？
neuron-charge 模块不区分。 它只管充电控制: 收到"开始充电"请求 → 检查 connector 状态 → 发 RemoteStart → 跟踪状态机 → 记录充电数据。

区分发生在上层调用方: 家庭用户的 API 接口直接调用 charge 模块; 企业用户的 API 接口先调用 billing 模块做预授权，再调用 charge 模块启动充电。

charge 和 billing 通过 Kafka 解耦: charge 模块在 StopTransaction 时发布 charging.completed 事件到 Kafka。billing 模块订阅该事件，只处理 BUSINESS 租户的订单。HOME 租户的事件被跳过。

可靠性设计

离线补偿与数据可靠性

数据分级

级别数据类型OCPP 消息离线处理
S级充电记录 / 计费数据StartTransaction / StopTransaction / MeterValues (带 txId)本地 Flash 持久化，恢复后按序补发

C级心跳 / DLM 状态Heartbeat / DataTransfer (DLMStatus)丢弃，恢复后重新开始
OCPP 的离线数据保障机制是什么？
OCPP 对交易相关消息 (Transaction-Related) 的保障是业务层的: 设备端必须持久化存储交易数据，直到收到云平台的 CALLRESULT 确认。即使 WebSocket 断开、云端重启，设备恢复连接后会自动补发所有未确认的交易消息。这是 OCPP 协议规范层面的硬性要求，保障的不只是"消息送到了"，而是"业务方确认收到了"。

S 级数据: 端到端零丢失链路

1
设备端 Flash 持久化 — 交易数据写入本地存储，断网不丢。OCPP 配置 TransactionMessageAttempts ≥ 3，TransactionMessageRetryInterval = 30s
2
OCPP CALL-CALLRESULT 确认 — 每条交易消息都有 UniqueId 匹配的响应，确保云平台已接收
3
Kafka 持久化 — 进入 Kafka 后 acks=all 多副本写入，Broker 宕机不丢数据
4
消费端幂等 — Redis 去重 (chargePointId + connectorId + timestamp)，重复消息不重复入库
5
对账机制 (预留) — 定时对比设备端和云端的交易序号，发现缺口触发补传

DLM 离线安全

DLM 完全在 N3 Lite 本地运行，不依赖云端。

断网前: 云平台通过 DataTransfer 下发 DLM 策略 → N3 Lite 存入 Flash (maxCurrent=63A, mode=balanced)
断网中: N3 Lite 本地 CT 采集 → 本地计算可分配电流 → PLC 下发各桩限值。云平台参与度 = 0
恢复后: N3 Lite 上报当前 DLM 分配状态 (DataTransfer)，云平台更新显示

OTA 远程升级

所有设备的 OTA 都通过 OCPP UpdateFirmware 消息触发:

设备OTA 通道说明

1
上传固件 → MinIO → 生成临时签名 URL (30 分钟过期)
2
创建升级任务 — 选择版本 + 目标设备 (单台/批量/灰度: 先 10% → 观察 24h → 全量)
3

5
结果上报 — FirmwareStatusNotification → 云端标记成功/失败

为什么不用 MQTT Broker？
所有设备都通过 OCPP 协议 (WebSocket) 接入同一个 Netty 服务，不需要独立的 MQTT Broker。少一个中间件 = 少一个运维组件 = 更简单的部署架构。如果未来有纯 MQTT 设备需要接入，可按需引入。

  • 智能充电 (可选) — 定时充电、谷电时段策略

交付物: 家庭用户可通过 App 远程控制充电、查看充电记录和用电统计。

Phase 3 — 企业运营 后续
  • 计费结算 (neuron-billing) — 电价策略、预授权、扣款
  • 运营商管理 (neuron-operator) — 入驻、站点、分账、收入报表
  • 租户类型扩展 (BUSINESS) — 扫码充电 App
  • 第三方支付对接 + 分账 (具体渠道根据目标市场确定)

交付物: 企业运营商入驻、公众用户扫码充电、按量计费、分账结算。

Phase 4 — 差异化 远期
  • 开放 API (neuron-openapi) — 第三方接入
  • 光储充一体化 (neuron-solar)
  • 电网互动 / V2G (neuron-grid)

交付物: 平台生态、高级能源管理。

设计原则总结

#原则为什么
1数据通道先行先打通设备连接和数据沉淀，再叠加充电控制，最后加入计费。每一层都建立在上一层稳定的基础上
2单协议统一 (OCPP)直连桩和 N3 Lite 代理桩对云平台完全透明，业务层只写一套代码。参考施耐德 EVlink Pro AC 集群做法
3N3 Lite 即桩虚拟为多枪充电桩，CT 数据用 connector 0 MeterValues (合规)，DLM 用 DataTransfer (标准扩展)
4关键数据零丢失OCPP 交易消息持久化 + Kafka acks=all + 消费端幂等，端到端零丢失
5离线安全N3 Lite 本地 DLM 和充电控制不依赖云端，LocalAuthList 离线鉴权，交易数据断网缓存恢复后补发
6模块化单体当前规模不需要微服务，模块边界 (Maven module) 保证未来拆分低成本
7双租户体系HOME 家庭用户 + BUSINESS 企业运营商，同一套系统按类型控制功能。家庭可一键升级为运营商
8计费以电表为准OCPP Start/StopTransaction 的 meterStart/meterStop 是唯一计费依据 (企业租户)
必须坚持
  • N3 Lite 的 OCPP 代理对云平台完全透明
  • 云端只下发配置和策略，不参与实时控制环路
  • 每层模块只依赖下层，不依赖上层 (device ← charge ← billing)
  • 所有 OCPP 消息 (含 DataTransfer) 统一转 Kafka
  • S 级数据全链路有保障
  • 计费以 OCPP 电表读数为唯一依据
必须避免
  • 在 charge 模块中写支付/计费逻辑 (应在 Phase 3 的 billing 模块)
  • 业务层区分"直连桩"和"N3 Lite"做不同逻辑
  • 引入不必要的协议和中间件 (当前不需要 MQTT Broker)
  • 设计假设设备永远在线的功能
  • 平台直接收款、沉淀用户资金
  • 过早拆分微服务
